# 巡检宝项目 - Code Review规范

> 本规范适用于巡检宝项目的代码审查活动。
> **核心原则**: 质量第一、建设性反馈、知识共享

---

## 一、Review流程

```
提交PR → 自动检查 → 人工Review → 修改 → 合并
   ↑                                    ↓
   └──────────── 不通过返回 ────────────┘
```

---

## 二、Review检查清单

### 2.1 功能性检查

- [ ] 代码是否符合需求
- [ ] 边界条件是否处理
- [ ] 错误处理是否完善
- [ ] 是否有明显Bug

### 2.2 安全性检查

- [ ] 是否有SQL注入风险
- [ ] 是否有XSS风险
- [ ] 敏感信息是否脱敏
- [ ] 权限检查是否到位

### 2.3 性能检查

- [ ] 是否有N+1查询
- [ ] 是否有内存泄漏
- [ ] 算法复杂度是否合理
- [ ] 资源是否正确释放

### 2.4 代码规范

- [ ] 命名是否清晰
- [ ] 注释是否必要
- [ ] 代码是否可测试
- [ ] 是否符合项目规范

---

## 三、Review原则

```yaml
✅ 应该做的:
  - 及时Review（24小时内）
  - 指出具体问题
  - 提供改进建议
  - 认可好的代码
  - 学习他人代码

❌ 不应该做的:
  - 人身攻击
  - 只挑错不肯定
  - 过度追求完美
  - 不说明原因的拒绝
  - 草率通过
```

---

## 四、Review评论规范

```
// ✅ 好的评论
"这里使用循环查询会导致N+1问题，建议使用Preload优化"

// ❌ 不好的评论
"这代码不行"

// ✅ 好的评论
"建议添加错误日志，方便排查问题"

// ❌ 不好的评论
"怎么没日志？"
```

---

## 五、合并标准

```yaml
必须满足:
  - 至少2人Review通过
  - 所有评论已解决
  - CI检查通过
  - 无冲突
  
特殊情况:
  - hotfix: 1人Review可合并
  - 文档修改: 1人Review可合并
```

---

## 六、禁止事项

```yaml
❌ 绝对禁止:
  - 未经Review直接合并
  - 强制推送覆盖他人提交
  - 合并未解决的冲突
  - 提交代码不关联Issue
```
