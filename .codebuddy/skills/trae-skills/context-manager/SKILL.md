# Context Manager - 上下文管理大师

> **核心使命**: 确保Agent在长时间对话中始终保持"头脑清醒"，不被冗长的上下文淹没 🎯

---

## 一、核心原则

```yaml
黄金法则:
  1️⃣ 上下文健康优先: 任何时候都要监控上下文长度
  2️⃣ 摘要优于堆积: 宁愿生成摘要，也不要让上下文无限增长
  3️⃣ 规则不能被淹没: 无论上下文多长，规则永远是最高优先级
  4️⃣ 主动预警: 不要等问题发生，要预防问题发生

健康标准:
  - L1 (0-50消息): 完美状态，畅快执行
  - L2 (50-100消息): 警戒状态，开始摘要
  - L3 (100-200消息): 警告状态，必须摘要
  - L4 (>200消息): 危险状态，暂停任务
  - L5 (>500消息): 崩溃状态，立即重置
```

## 二、自动监控机制

### 2.1 上下文长度实时监控

**每次响应前必须执行：**

```markdown
🔍 上下文健康检查

当前状态：
- 消息总数：87 条
- Token估算：约 25,000
- 当前级别：L2 警戒 ⚠️

应对措施：
✅ 上下文健康，可以继续
⚠️ 开始主动摘要，准备清理冗余
🔴 必须摘要后才能继续
🚨 暂停所有任务，立即摘要
```

### 2.2 自动触发摘要的条件

**必须立即摘要的情况：**

| 条件 | 级别 | 触发时机 |
|------|------|---------|
| 连续3次响应质量下降 | L2 | 自动触发 |
| 消息数超过100条 | L3 | 强制摘要 |
| 规则检查连续失败2次 | L2 | 必须摘要 |
| 任务完成阶段 | 任意 | 推荐摘要 |
| 用户明确要求 | 任意 | 立即执行 |

## 三、摘要生成器

### 3.1 摘要生成模板

当需要摘要时，输出以下格式：

```markdown
# 📋 对话摘要 - [时间戳]

## 任务目标 🎯
[用户最初想要完成的目标是什么]

## 已完成工作 ✅
1. ✅ [完成的任务1] - [简短说明]
2. ✅ [完成的任务2] - [简短说明]
3. ✅ [完成的任务3] - [简短说明]

## 当前状态 📍
- 正在执行：[当前正在做的任务]
- 进度：[百分比/阶段，如 60% / Phase 2/3]
- 遇到的问题：[如果有，描述问题和解决方案]
- 性能状态：[L1-L5 级别]

## 关键信息 🔑
- **涉及文件**：[主要修改的文件列表]
- **技术决策**：[做出的重要技术选择]
- **用户约定**：[对用户的承诺或约定]
- **依赖关系**：[前置条件或依赖项]

## 待完成任务 📝
- [ ] 任务A - [预期动作]
- [ ] 任务B - [预期动作]
- [ ] 任务C - [预期动作]

## 规则遵守情况 🛡️
- [✅] 所有技术信息已标注来源
- [✅] 关键操作已获得确认
- [✅] 推测内容已明确标注
- [✅] 错误已被纠正并说明

## 重新锚定确认 ✅
我已经重新阅读了所有相关规则：
- [✅] Agent幻觉防范规则
- [✅] 上下文长度管理规则
- [✅] 项目开发规范
- [✅] 当前任务相关规范

## 下一步行动 ▶️
基于以上摘要，我将：
1. [具体动作1]
2. [具体动作2]
3. [具体动作3]

---
💡 **摘要生成完成，上下文已清理，可以继续高效执行！**
```

### 3.2 选择性摘要策略

**不是所有内容都需要保留：**

```yaml
保留的内容:
  ✅ 任务目标和关键决策
  ✅ 技术方案和设计选择
  ✅ 涉及的文件和代码片段
  ✅ 用户的重要反馈和要求
  ✅ 未完成的工作清单
  ✅ 规则遵守状态

可以丢弃的内容:
  ❌ 重复的解释和说明
  ❌ 失败的操作尝试（除非有重要教训）
  ❌ 过时的技术方案
  ❌ 无关的闲聊内容
  ❌ 已经完成的测试输出
  ❌ 调试用的临时信息
```

## 四、规则守护程序 🛡️

### 4.1 守护程序检查清单

**每5个操作必须执行一次：**

```markdown
🛡️ 规则守护程序检查

## 1. 幻觉防范检查 🔍
- [ ] 最近提供的信息是否都有来源标注？
- [ ] 是否有未标注的推测内容？
- [ ] 是否有无法验证的断言？
- [ ] 关键操作是否已获得用户确认？

## 2. 上下文长度检查 📏
- [ ] 当前上下文长度级别？
- [ ] 是否需要生成摘要？
- [ ] 是否有信息丢失的风险？

## 3. 任务进度检查 📊
- [ ] 是否偏离原始目标？
- [ ] 是否需要重新确认用户意图？
- [ ] 是否遇到超出能力范围的问题？

## 4. 规则偏离检查 ⚠️
- [ ] 是否有任何规则被忽略？
- [ ] 是否有任何简化步骤被跳过？
- [ ] 是否有任何"为了方便"而违反规则的行为？

## 5. 性能状态检查 🚀
- [ ] 最近3次响应是否流畅？
- [ ] 是否有重复思考或循环？
- [ ] 是否需要优化工作流程？

---

检查结果：
- 幻觉防范：✅ 通过 / ⚠️ 警告
- 上下文健康：✅ 正常 / ⚠️ 需要摘要
- 任务进度：✅ 正常 / ⚠️ 偏离
- 规则遵守：✅ 良好 / ⚠️ 有偏离
- 性能状态：✅ 高效 / ⚠️ 下降

总体评估：✅ 可以继续 / ⚠️ 需要调整 / 🔴 需要暂停
```

### 4.2 紧急标志 🚨

**如果以下任一条件为真，立即停止并报告：**

```yaml
🚨 紧急停止条件:
  - 上下文达到 L4 危险级别 (>200消息)
  - 发现可能导致数据丢失的操作
  - 发现可能违反安全规则的操作
  - 连续 3 次规则守护检查失败
  - 无法理解用户意图或任务要求
  - 感到困惑或严重不确定

立即行动:
  1. 停止当前操作
  2. 生成紧急摘要
  3. 向用户报告问题
  4. 请求进一步指导
```

## 五、主动预警系统 ⚠️

### 5.1 性能下降预警

**当检测到以下迹象时，主动预警：**

```yaml
预警信号:
  - 连续3次响应时间超过预期
  - 频繁需要重新读取文件
  - 开始重复相同的问题
  - 越来越难以理解上下文
  - 规则检查通过率下降

预警模板:
  ```markdown
  ⚠️ 性能预警

  我注意到上下文正在变得冗长，这可能导致我的表现下降。

  当前状态：
  - 上下文长度：L2 警戒
  - 性能指标：⚠️ 轻微下降

  建议措施：
  1. [ ] 生成对话摘要
  2. [ ] 确认关键信息
  3. [ ] 清理冗余内容

  您希望我现在生成摘要吗？这将帮助我们更高效地继续工作。
  ```
```

### 5.2 任务复杂度预警

**当任务复杂度超过阈值时：**

```markdown
🎯 复杂度预警

当前任务评估：
- 文件数量：15 个 ⚠️
- 代码行数：1500 行 ⚠️
- 依赖数量：7 个 ⚠️
- 不确定性：中 ⚠️

综合复杂度：高 ⚠️

建议：
1. 将任务分解为多个子任务
2. 分批完成，每批后生成摘要
3. 每个关键节点重新确认规则

您希望我如何继续？
```

## 六、紧急恢复机制 🚨

### 6.1 规则偏离纠正

**当检测到规则偏离时：**

```markdown
🚨 规则偏离检测

我发现之前的响应违反了规则：

偏离内容：
❌ [错误行为描述]

规则要求：
✅ [正确行为描述]

立即纠正：
✅ [已经采取的纠正措施]

防止再次偏离：
1. [ ] 设置提醒
2. [ ] 重新阅读相关规则
3. [ ] 记录到经验库

感谢您的耐心，我将继续严格执行规则。
```

### 6.2 上下文损坏恢复

**当上下文可能损坏时：**

```markdown
🔄 上下文损坏检测

可能的问题：
- 信息丢失或不连贯
- 无法访问之前的上下文
- 重复询问相同问题

建议行动：
1. 生成当前状态摘要
2. 重新确认关键信息
3. 清理并重建上下文

您是否可以提供以下关键信息：
1. 当前任务的主要目标？
2. 最重要的已完成工作？
3. 当前遇到的主要问题？

这将帮助我快速恢复上下文。
```

## 七、使用指南 📖

### 7.1 何时调用此Skill

**自动触发（无需用户要求）：**
- 开始任何开发任务时
- 检测到上下文长度超过 L2 级别时
- 每执行 5 个操作后
- 任务完成或进入新阶段时
- 规则守护检查时

**用户触发：**
- 用户说"检查上下文"或"上下文健康"
- 用户说"生成摘要"
- 用户说"感觉变慢了"
- 用户要求"清理上下文"

### 7.2 执行流程

```
用户请求
    │
    ├─ 立即检查上下文长度
    │
    ├─ 如果 L1:
    │   └─ 继续执行，无需干预
    │
    ├─ 如果 L2:
    │   ├─ 提醒用户上下文开始变长
    │   ├─ 准备摘要
    │   └─ 继续执行，但增加检查频率
    │
    ├─ 如果 L3:
    │   ├─ 必须生成摘要
    │   ├─ 获得用户确认
    │   └─ 清理后继续
    │
    └─ 如果 L4/L5:
        ├─ 暂停所有任务
        ├─ 生成紧急摘要
        └─ 请求用户确认如何继续
```

### 7.3 与其他Skill的协作

```yaml
协作Skill:
  - skill-dispatcher: 任务分析后，调用context-manager评估任务复杂度
  - global-dev: 开发过程中，定期调用守护程序检查
  - code-review: 审查前，确保上下文健康
  - bug-fix: Bug修复后，生成摘要记录

协作时机:
  - 任务开始 → context-manager评估复杂度
  - 任务进行中 → 每5个操作检查一次
  - 任务完成 → context-manager生成摘要
  - 任务切换 → context-manager清理上下文
```

## 九、最佳实践 🌟

### 9.1 预防优于治疗

```yaml
预防措施:
  ✅ 任务开始前评估复杂度
  ✅ 设置检查点
  ✅ 定期生成摘要
  ✅ 及时清理冗余
  ✅ 保持规则意识

避免情况:
  ❌ 等到上下文崩溃才处理
  ❌ 忽略性能下降信号
  ❌ 为了方便跳过规则检查
  ❌ 积累大量未整理的信息
```

### 9.2 摘要质量标准

```yaml
优秀摘要的特征:
  ✅ 简洁明了，不超过500字
  ✅ 覆盖所有关键信息
  ✅ 清晰的任务状态
  ✅ 明确的下一步行动
  ✅ 规则遵守确认

糟糕摘要的特征:
  ❌ 冗长复杂，超过1000字
  ❌ 遗漏关键信息
  ❌ 任务状态模糊
  ❌ 缺乏行动指导
```

### 9.3 与用户协作

```yaml
用户可以提供帮助:
  ✅ 确认任务目标
  ✅ 提供优先级指导
  ✅ 确认关键决策
  ✅ 反馈工作质量
  ✅ 授权清理冗余

用户应该知道:
  📊 当前的上下文健康状态
  ⚠️ 任何性能下降
  🎯 任务进度和状态
  📝 即将执行的摘要操作
```

## 十、总结 📝

**Context Manager 的核心职责：**

1. 🎯 **保持上下文健康** - 监控长度，定期清理
2. 🛡️ **守护规则执行** - 防止规则被淹没
3. ⚡ **保持高效执行** - 预防性能下降
4. 🚨 **主动预警** - 不等问题发生
5. 🔄 **快速恢复** - 从问题中快速恢复

**记住：**
- 上下文就像手机电量，用一点少一点
- 摘要就像充电宝，让你的"脑子"重新满血
- 规则守护就像安全带，关键时刻保护你
