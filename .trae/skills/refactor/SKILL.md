# Refactor - 重构大师

> **核心使命**: 让代码越来越优雅，重构而不改变行为！🔨

---

## 一、核心原则

```yaml
黄金法则:
  1️⃣ 测试护航: 没有测试不要重构
  2️⃣ 小步前进: 每次只改一点
  3️⃣ 行为不变: 重构不改变功能
  4️⃣ 持续改进: 重构是日常工作

重构前提:
  ✅ 有测试覆盖
  ✅ 功能正常工作
  ✅ 理解代码意图
  ✅ 制定重构计划

重构禁忌:
  ❌ 没有测试就重构
  ❌ 边重构边改功能
  ❌ 一次性大改
  ❌ 不理解代码就重构
```

   - 函数长度
   - 耦合度

2. 风险评估
   - 影响范围
   - 测试覆盖率
   - 团队熟悉度
   - 依赖复杂度

3. 收益评估
   - 可维护性提升
   - 性能改善
   - 开发效率提升
   - 技术债务减少

评估模板:
  ## 📊 重构评估报告

  ### 代码质量
  - 复杂度: B级
  - 重复率: 25%
  - 覆盖率: 60%

  ### 风险等级: 中

**计划内容**：

     - 涉及哪些文件
     - 修改哪些模块

  📝 步骤清单
     - 步骤1: [操作]
     - 步骤2: [操作]

  ⏰ 里程碑
     - 里程碑1: [完成标志]
     - 里程碑2: [完成标志]

  🛡️ 回滚计划
     - 回滚点: [时间点]
     - 回滚方法: [操作]

### 2.3 安全重构执行

**执行原则**：

```yaml
安全重构原则:
  1️⃣ 红-绿-重构
     - Red: 写一个失败测试
     - Green: 最快速度让测试通过
     - Refactor: 重构代码
  
  2️⃣ 三步法
     - 第一步: 添加新代码
     - 第二步: 运行测试
     - 第三步: 删除旧代码
  
  3️⃣ 童子军规则
     - 每次离开时，代码比来时更干净
  
  4️⃣ 适时重构
     - 不要大范围重构
     - 在添加功能时重构
     - 在修复Bug时重构
```

## 三、坏味道识别

### 3.1 代码层面

```yaml
味道1: 重复代码 (Duplicated Code)
症状:
  - 相同或相似的代码出现3次以上
  - 复制粘贴的痕迹
  - 修改一个地方需要改多处

处理:
  - 提取为函数
  - 提取为基类
  - 提取为常量
  - 使用模板方法模式

味道2: 过长函数 (Long Method)
症状:
  - 函数超过50行
  - 函数有多个注释分隔的部分
  - 参数过多

处理:
  - 提取函数
  - 提取类
  - 使用解释性变量
  - 用策略替换条件

味道3: 过大类 (Large Class)
症状:
  - 类超过300行
  - 类的属性过多
  - 类的职责过多

处理:
  - 提取子类
  - 提取接口
  - 使用组合
  - 拆分大类
```

### 3.2 设计层面

```yaml
味道4: 霰弹式修改 (Shotgun Surgery)
症状:
  - 修改一个功能需要改很多文件
  - 修改分散在各处
  - 改一个Bug引发另一个Bug

处理:
  - 移动方法到正确的类
  - 移动字段到正确的类
  - 提取内聚模块
  - 使用中介者模式

味道5: 特性依恋 (Feature Envy)
症状:
  - 一个类的方法频繁使用另一个类的数据
  - 方法属于另一个类

处理:
  - 移动方法到数据所在类
  - 提取方法
  - 使用设计模式

味道6: 数据泥团 (Data Clumps)
症状:
  - 几个字段总是同时出现
  - 参数列表总是有相同的几个参数

处理:
  - 提取为类
  - 提取为参数对象
  - 保持完整
```

### 3.3 架构层面

```yaml
味道7: 神秘命名 (Mysterious Name)
症状:
  - 变量名无法描述其用途
  - 函数名无法描述其行为
  - 类名过于笼统

处理:
  - 重命名
  - 澄清意图
  - 使用业务语言

味道8: 冗余元素 (Redundant Element)
症状:
  - 无用的注释
  - 无用的代码
  - 重复的抽象

处理:
  - 删除注释
  - 删除死代码
  - 合并相似抽象

味道9: 夸夸其谈的未来性
症状:
  - 过度设计
  - 不必要的抽象
  - 暂时用不到的功能

处理:
  - 删除不需要的抽象
  - YAGNI原则
  - 简化设计
```

## 四、重构技术库

### 4.1 提取技术

```yaml
提取函数:
  // Before
  function processOrder(order) {
    // 计算总价
    let total = 0
    for (let item of order.items) {
      total += item.price * item.quantity
    }
    // 应用折扣
    if (total > 100) {
      total *= 0.9
    }
    // 保存订单
    saveOrder(order)
  }

  // After
  function processOrder(order) {
    const total = calculateTotal(order)
    const discounted = applyDiscount(total)
    saveOrder(order, discounted)
  }

提取类:
  // Before
  class User {
    name, email, phone
    validateEmail(), validatePhone()
    sendEmail(), sendSMS()
  }

  // After
  class User {
    name, email, phone
  }
  class EmailService {
    validate(), send()
  }
  class SMSService {
    validate(), send()
  }
```

### 4.2 搬移技术

```yaml
搬移方法:
  // Before
  class Account {
    charge() {
      // 计算利息
    }
  }

  // After
  class Account {
    charge() {
      InterestCalculator.calculate(this)
    }
  }

搬移字段:
  // Before
  class Order {
    customerName  // 在Order中
    customerEmail
  }

  // After
  class Order {
    customer  // 引用Customer对象
  }
  class Customer {
    name, email
  }
```

### 4.3 重新组织技术

```yaml
重命名:
  // Before
  function proc() {}
  const d = new Date()

  // After
  function processOrder() {}
  const orderDate = new Date()

提炼接口:
  // Before
  class UserService {
    createUser()
    updateUser()
    deleteUser()
    getUserById()
  }

  // After
  interface UserReader {
    getUserById()
  }
  interface UserWriter {
    createUser()
    updateUser()
    deleteUser()
  }
```

## 六、重构策略

### 6.1 渐进式重构

```yaml
策略1: 童子军规则
  - 每次修改代码时
  - 让代码比来时更干净
  - 小步累积，长期受益

策略2: 趁热重构
  - 在添加新功能时重构
  - 在修复Bug时重构
  - 在代码审查时重构

策略3: 零散重构
  - 不要大范围重构
  - 每次只重构一小块
  - 逐步改善代码库
```

### 6.2 性能重构

```yaml
时机选择:
  ✅ 性能明显问题时
  ✅ 有性能测试基准时
  ✅ 充分测试覆盖时
  ✅ 理解性能影响时

策略:
  ✅ 先测量再优化
  ✅ 找到真正的瓶颈
  ✅ 小步优化验证
  ✅ 避免过早优化
```

## 十、总结 📝

**Refactor 核心职责**：

1. 🔍 **坏味道识别** - 发现需要重构的代码
2. 📋 **安全重构** - 有计划、有测试的重构
3. 🛡️ **行为不变** - 重构不改变功能
4. ✅ **持续改进** - 逐步提升代码质量
5. 📚 **经验积累** - 记录重构经验

**记住**：
- 重构是日常工作，不是专门的大工程
- 测试是重构的保障，没有测试不要重构
- 小步前进，每次只改一点
- 重构不改变行为，改变行为是另一件事
