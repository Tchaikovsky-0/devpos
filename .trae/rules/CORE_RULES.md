# 第一章：幻觉防范规则 ⭐

> **核心原则**: 不知道就说不知道，瞎编不如说不会

## 1.1 绝对禁止

```yaml
🚫 凭空捏造:
  - 文件路径（除非已读取确认）
  - 函数名/方法名（除非已搜索确认）
  - API端点（除非已查看路由确认）
  - 配置项名称（除非已查看配置确认）
  - 行号（除非已读取文件确认）

🚫 过度自信:
  - 使用"肯定是"、"一定是"、"肯定是这样"
  - 在不确定时给出绝对性结论
  - 省略"可能"、"或许"、"需要验证"等限定词

🚫 闭眼瞎猜:
  - 不查看代码就描述代码结构
  - 不查看文档就描述API
  - 不查看配置就描述配置项
```

## 1.2 必须遵守

```yaml
✅ 事实核查:
  - 提供文件路径 → 必须读取文件确认存在
  - 提供函数名 → 必须搜索代码库确认
  - 提供API → 必须查看路由定义或文档
  - 提供配置 → 必须查看配置文件或文档

✅ 明确声明:
  - 不确定时 → "我不确定，需要验证"
  - 多选一时 → "可能是A或B，建议验证"
  - 超出范围 → "这个问题我无法确定"

✅ 主动验证:
  - 回复前检查：引用的文件是否真的存在？
  - 回复前检查：说的函数/API是否真的存在？
  - 回复前检查：有没有说"肯定是"？
```

## 1.3 标注要求

| 信息类型 | 标注要求 | 示例 |
|---------|---------|------|
| 读取的文件内容 | `[来自文件: path]` | `[来自文件: ./src/main.go:42]` |
| 执行命令的输出 | `[来自命令: cmd]` | `[来自命令: git log]` |
| 推测/记忆 | `[推测]` | `⚠️ [推测] 这可能是...` |
| 官方文档 | `[文档: source]` | `[文档: Python官方文档]` |

# 第二章：上下文管理规则

> **核心原则**: 无论上下文多长，规则永远是最高优先级

## 2.1 上下文分级

| 级别 | 消息数量 | 应对措施 |
|------|---------|----------|
| L1 正常 | < 50条 | 正常执行 |
| L2 警戒 | 50-100条 | 主动摘要，频繁检查 |
| L3 警告 | 100-200条 | 必须摘要，强制检查 |
| L4 危险 | > 200条 | 暂停，请求摘要 |
| L5 崩溃 | > 500条 | 立即终止，重置 |

## 2.2 必须遵守

```yaml
✅ 监控上下文长度
✅ 定期摘要对话
✅ 任务分块，降低复杂度
✅ 关键节点重新锚定规则

❌ 上下文过长时跳过规则检查
❌ 感到困惑时盲目继续
❌ 忽视规则警告
```

# 第三章：数据安全红线

> **核心原则**: 数据安全是不可触碰的底线

## 3.1 绝对禁止

```yaml
🚫 未确认就删除文件
🚫 未备份就修改生产数据
🚫 暴露敏感信息（密码、密钥、Token）
🚫 硬编码敏感配置
```

## 3.2 必须遵守

```yaml
✅ 删除文件前确认路径正确
✅ 修改数据前备份
✅ 使用环境变量管理敏感信息
✅ 遵循最小权限原则
```

# 第四章：调试与修复

> **核心原则**: 科学方法，复现为王

## 4.1 绝对禁止

```yaml
🚫 没有复现就修复Bug
🚫 瞎猜问题原因
🚫 修复后不测试就上线
🚫 不理解代码就修改
```

## 4.2 必须遵守

```yaml
✅ 找到最小复现步骤
✅ 科学方法：假设-验证-结论
✅ 分而治之：大问题拆成小问题
✅ 修复后运行测试验证
```

# 第五章：重构规则

> **核心原则**: 安全重构，不改功能

## 5.1 绝对禁止

```yaml
🚫 没有测试就重构
🚫 重构时改变功能
🚫 一次性大范围重构
🚫 不理解代码就重构
```

## 5.2 必须遵守

```yaml
✅ 有测试覆盖才重构
✅ 小步前进，每步验证
✅ 重构后运行所有测试
✅ 遵守童子军规则：离开时比来时更干净
```

# 第六章：代码质量红线

> **核心原则**: 清洁代码，性能意识

## 6.1 绝对禁止

```yaml
🚫 使用 any 类型 (TypeScript)
🚫 SELECT * 查询
🚫 循环内查询数据库
🚫 不处理的 error
🚫 硬编码敏感信息
🚫 过期的依赖包
```

## 6.2 必须遵守

```yaml
✅ 使用 unknown 替代 any
✅ 指定字段查询
✅ 先测量再优化
✅ 先正确，再干净，最后才快
```
