# 代码审查清单与机制

## 巡检宝代码审查指南

> **目的**: 确保代码质量、防止技术债务积累、提升团队协作效率
> 
> **适用范围**: 所有代码提交、PR合并、功能发布
> 
> **最后更新**: 2026年4月

---

## 一、代码审查流程

### 1.1 审查时机

| 场景 | 审查要求 | 审查人 |
|------|---------|--------|
| **功能开发** | 必须审查 | 至少1名团队成员 |
| **Bug修复** | 必须审查 | 至少1名团队成员 |
| **代码重构** | 必须审查 | 至少1名高级成员 |
| **紧急修复** | 事后审查 | 至少1名团队成员 |
| **文档更新** | 快速审查 | 可选 |

### 1.2 审查级别

#### **Level 1: 快速审查** (< 10分钟)
适用于：
- 小改动 (< 50行代码)
- 文档更新
- 配置修改
- Bug快速修复

审查重点：
- ✅ 代码逻辑正确性
- ✅ 无明显错误
- ✅ 符合命名规范

#### **Level 2: 标准审查** (10-30分钟)
适用于：
- 中等改动 (50-300行代码)
- 新功能开发
- 组件重构

审查重点：
- ✅ 所有Level 1检查项
- ✅ 代码质量和架构
- ✅ 性能和安全性
- ✅ 测试覆盖

#### **Level 3: 深度审查** (30-60分钟)
适用于：
- 大改动 (> 300行代码)
- 架构变更
- 核心模块修改

审查重点：
- ✅ 所有Level 2检查项
- ✅ 系统架构设计
- ✅ API设计合理性
- ✅ 扩展性和维护性

## 二、代码审查清单

### 2.1 功能正确性

- [ ] **业务逻辑**: 代码逻辑是否符合业务需求？
  - 实现的功能点完整
  - 边界条件处理正确
  - 错误处理完善

- [ ] **数据处理**: 数据处理是否正确？
  - 数据类型转换正确
  - 数据验证完善
  - 空值处理完善

- [ ] **API调用**: API调用是否正确？
  - 请求参数正确
  - 响应处理完善
  - 错误处理合理

### 2.2 代码质量

- [ ] **代码规范**: 代码是否符合项目规范？
  - [ ] ESLint规则通过
  - [ ] TypeScript类型正确
  - [ ] 命名规范符合
  - [ ] 代码风格一致

- [ ] **代码复杂度**: 代码复杂度是否合理？
  - [ ] 函数长度 < 50行
  - [ ] 文件长度 < 500行
  - [ ] 圈复杂度 < 10
  - [ ] 无深层嵌套 (>3层)

- [ ] **代码重复**: 是否存在代码重复？
  - [ ] 无重复代码
  - [ ] 重复代码已抽象
  - [ ] 复用现有组件

### 2.3 组件设计

- [ ] **组件划分**: 组件划分是否合理？
  - [ ] 单一职责原则
  - [ ] 组件粒度适中
  - [ ] 无上帝组件

- [ ] **组件复用**: 组件是否可复用？
  - [ ] Props设计合理
  - [ ] 样式与逻辑分离
  - [ ] 无硬编码值

- [ ] **组件测试**: 组件是否可测试？
  - [ ] 无内部状态依赖
  - [ ] 依赖注入设计
  - [ ] 副作用最小化

### 2.4 性能优化

- [ ] **渲染性能**: 是否避免不必要的渲染？
  - [ ] React.memo使用合理
  - [ ] useMemo/useCallback使用恰当
  - [ ] 列表渲染使用key

- [ ] **数据获取**: 数据获取是否优化？
  - [ ] 无N+1问题
  - [ ] 缓存策略合理
  - [ ] 懒加载使用恰当

- [ ] **Bundle优化**: 是否避免大Bundle？
  - [ ] 代码分割使用
  - [ ] 动态导入使用
  - [ ] 无不必要的依赖

### 2.5 安全性

- [ ] **输入验证**: 输入验证是否完善？
  - [ ] 所有用户输入验证
  - [ ] 类型检查完善
  - [ ] SQL注入防护

- [ ] **敏感信息**: 敏感信息是否保护？
  - [ ] 无硬编码密钥
  - [ ] 环境变量使用
  - [ ] 日志脱敏处理

- [ ] **XSS防护**: XSS防护是否完善？
  - [ ] 输出转义
  - [ ] CSP配置正确
  - [ ] DOM操作安全

### 2.6 可访问性

- [ ] **语义化**: HTML语义化是否正确？
  - [ ] 正确的标签使用
  - [ ] 标题层级合理
  - [ ] 区域划分清晰

- [ ] **键盘操作**: 键盘操作支持？
  - [ ] Tab顺序合理
  - [ ] 快捷键支持
  - [ ] 焦点管理正确

- [ ] **屏幕阅读**: 屏幕阅读支持？
  - [ ] ARIA标签使用
  - [ ] 图片alt文本
  - [ ] 链接文本描述

### 2.7 测试覆盖

- [ ] **单元测试**: 单元测试是否完善？
  - [ ] 覆盖率 > 80%
  - [ ] 边界条件覆盖
  - [ ] 错误场景覆盖

- [ ] **集成测试**: 集成测试是否完善？
  - [ ] API集成测试
  - [ ] 组件集成测试
  - [ ] 跨组件交互

- [ ] **E2E测试**: E2E测试是否完善？
  - [ ] 关键路径覆盖
  - [ ] 用户场景测试
  - [ ] 性能测试

## 三、审查反馈指南

### 3.1 反馈优先级

| 标记 | 含义 | 行动 |
|------|------|------|
| **[必须修复]** | 严重问题 | PR不能合并 |
| **[建议修复]** | 中等问题 | 建议修复，可讨论 |
| **[可选优化]** | 轻微问题 | 可接受本次合并 |
| **[提问]** | 需要澄清 | 请作者说明 |

### 3.2 反馈模板

#### **必须修复 (Must Fix)**
```
[必须修复] #行号 - 问题描述

当前代码:
```tsx
// 问题代码
```

建议修改:
```tsx
// 建议修改
```

原因: 解释为什么需要修改
```

#### **建议修复 (Should Fix)**
```
[建议修复] #行号 - 问题描述

当前代码:
```tsx
// 问题代码
```

建议修改:
```tsx
// 建议修改
```

可选方案: 其他可能的解决方案
```

#### **可选优化 (Could Improve)**
```
[可选优化] #行号 - 优化建议

当前代码:
```tsx
// 当前代码
```

优化建议:
```tsx
// 优化后的代码
```

收益: 优化带来的好处
```

## 四、审查节奏

### 4.1 PR创建要求

- [ ] PR标题清晰描述改动
- [ ] PR描述包含改动原因
- [ ] 关联的Issue已链接
- [ ] 测试已添加/更新
- [ ] 文档已更新（如需要）

### 4.2 审查时间要求

| PR大小 | 审查时间 | 最大时间 |
|--------|---------|---------|
| **小** (< 50行) | < 10分钟 | 4小时 |
| **中** (50-300行) | < 30分钟 | 24小时 |
| **大** (> 300行) | < 60分钟 | 48小时 |

### 4.3 合并要求

#### **必须满足**
- ✅ 所有审查反馈已处理
- ✅ CI/CD流水线通过
- ✅ 至少1人批准
- ✅ 代码冲突已解决

#### **建议满足**
- 💡 测试覆盖率达标
- 💡 文档已完善
- 💡 无重大性能影响

## 五、持续改进

### 5.1 审查数据收集

每月收集以下数据：
- 平均审查时间
- PR合并率
- 常见问题分类
- 重复出现问题

### 5.2 问题分类

| 类别 | 说明 | 预防措施 |
|------|------|---------|
| **逻辑错误** | 业务逻辑问题 | 加强业务理解 |
| **代码规范** | 风格不一致 | 加强规范培训 |
| **性能问题** | 性能不达标 | 性能意识培训 |
| **安全问题** | 安全漏洞 | 安全编码规范 |
| **测试缺失** | 测试不足 | 测试覆盖率要求 |

### 5.3 改进计划

每季度进行一次审查流程回顾：
1. 分析审查数据
2. 识别常见问题
3. 制定改进措施
4. 实施并跟踪效果

## 七、附录

### A. 常见问题案例

#### 案例1: 硬编码敏感信息
```tsx
// ❌ 错误
const API_KEY = 'sk-1234567890abcdef';

// ✅ 正确
const API_KEY = process.env.REACT_APP_API_KEY;
```

#### 案例2: 不必要的重渲染
```tsx
// ❌ 错误
const handleClick = () => {
  setCount(count + 1);
};

// ✅ 正确
const handleClick = useCallback(() => {
  setCount(prev => prev + 1);
}, []);
```

#### 案例3: 缺少错误处理
```tsx
// ❌ 错误
const data = await fetch(url);

// ✅ 正确
try {
  const response = await fetch(url);
  if (!response.ok) throw new Error('Network response was not ok');
  const data = await response.json();
} catch (error) {
  console.error('Fetch error:', error);
  // 错误处理逻辑
}
```
