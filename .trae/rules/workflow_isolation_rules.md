# 巡检宝 - 多工作流隔离规则

> **版本**: v1.0.0
> **创建日期**: 2026-04-06
> **核心原则**: 隔离为王，互不干扰，安全并行

---

## 零、为什么需要工作流隔离？

想象一下：你在厨房做饭（工作流A），室友也在厨房做饭（工作流B）。
没有隔离 = 两人抢同一口锅、把对方的盐当成自己的、甚至把对方的菜给倒了。
有了隔离 = 各用各的灶台、各吃各的食材、偶尔借用调料也要打声招呼。

**多工作流并行开发时，最怕的就是：**
- 两个 Agent 同时改同一个文件 → 后写的覆盖先写的
- 两个工作流抢同一个端口 → 谁也跑不起来
- 一个工作流的 Bug 影响另一个 → 连环翻车
- 数据库被同时改表结构 → 数据丢失

**本规则就是为了防止这些灾难发生。**

## 一、工作流命名规范

### 1.1 命名格式

```
wf-{domain}-{feature}-{sequence}
```

| 字段 | 说明 | 示例值 |
|------|------|--------|
| `wf` | 固定前缀，标识这是一个工作流 | `wf` |
| `{domain}` | 所属技术领域 | `fe`、`be`、`ai`、`yolo`、`devops` |
| `{feature}` | 功能简述（kebab-case） | `user-auth`、`stream-mgmt`、`fire-detect` |
| `{sequence}` | 序列号（3位数字） | `001`、`002`、`003` |

### 1.2 命名示例

```yaml
合法命名:
  - wf-fe-login-page-001          # 前端登录页面开发
  - wf-be-user-api-002            # 后端用户API开发
  - wf-ai-detection-003           # AI检测功能开发
  - wf-yolo-fire-model-004        # YOLO火灾模型训练
  - wf-devops-ci-pipeline-005     # CI/CD流水线搭建

非法命名:
  - login                          # 缺少前缀和领域
  - workflow-fe-login              # 前缀错误
  - wf-fe-Login-Page-001          # 必须小写
  - wf_fe_login_001               # 必须用连字符
```

### 1.3 领域（domain）定义

| domain | 说明 | 涉及目录 | 负责 Agent |
|--------|------|----------|------------|
| `fe` | 前端 | `frontend/` | `frontend-dev`、`frontend-lead` |
| `be` | 后端 | `backend/` | `backend-dev`、`backend-lead` |
| `ai` | AI 服务 | `ai-service/` | `ai-lead`、`openclaw-eng` |
| `yolo` | YOLO 检测 | `yolo-service/` | `ai-lead` |
| `devops` | 运维部署 | `deploy/`、`scripts/`、`.github/` | `devops-eng` |
| `cross` | 跨领域 | 多个目录 | `project-lead` |

### 1.4 工作流元数据

每个工作流必须声明元数据，作为"身份证"：

```yaml
wf-be-user-api-002:
  name: "用户管理API"
  domain: be
  owner: backend-dev
  reviewer: backend-lead
  created_at: "2026-04-06T10:00:00Z"
  status: active                    # active | paused | completed | cancelled
  priority: P1                      # P0-P3
  scope:
    directories:
      - backend/internal/handler/
      - backend/internal/service/
      - backend/internal/repository/
      - backend/internal/model/
    files: []
    exclude:
      - backend/internal/middleware/   # 明确排除的目录
  resources:
    ports: [8094]                     # 需要的端口
    services: [mysql, redis]           # 需要的服务
  dependencies: []                    # 依赖的其他工作流
  conflict_zones:                     # 可能的冲突区域
    - backend/internal/model/user.go
    - backend/internal/router/router.go
```

## 二、资源分配机制

### 2.1 资源类型

```yaml
资源分类:
  硬性资源:
    - 端口（Port）           # 不可共享，独占
    - 文件（File）           # 写操作独占，读操作共享
    - 数据库表（Table）      # DDL 独占，DML 按行隔离

  软性资源:
    - 内存（Memory）        # 有限制，可超额申请
    - CPU（Compute）        # 有限制，可超额申请
    - 终端（Terminal）      # 独占，最多5个
```

### 2.2 端口分配表

> **核心原则**: 端口是独占资源，一个端口同一时间只能被一个工作流使用。

```yaml
端口分配:
  生产端口（不可占用）:
    3000: "前端生产"
    8094: "后端API"
    8095: "AI服务"
    8096: "OpenClaw"
    8097: "YOLO服务"
    3306: "MySQL"
    6379: "Redis"

  开发端口池（工作流可申请）:
    5100-5199: "前端开发工作流"
    8200-8299: "后端开发工作流"
    8300-8399: "AI开发工作流"
    8400-8499: "YOLO开发工作流"
    9000-9099: "临时工具/调试"

分配规则:
  - 工作流启动时从对应端口池申请
  - 先到先得，用完释放
  - 工作流结束后必须释放端口
  - 检测端口占用: lsof -i :{port}
```

### 2.3 终端分配

> **核心原则**: 最多5个终端，工作流结束时必须释放。

```yaml
终端分配:
  限制: 最多5个终端同时运行

  分配策略:
    - 每个工作流最多占用1个终端
    - P0 级别任务可抢占低优先级终端
    - 长时间空闲（>10分钟）的终端自动释放

  使用规范:
    - 终端启动时标注工作流ID
    - 阻塞命令和异步命令明确区分
    - 退出时必须清理临时进程
```

### 2.4 文件锁机制

```yaml
文件锁规则:
  读锁（共享锁）:
    - 多个工作流可同时读取同一文件
    - 不阻塞其他读操作
    - 读取前无需申请

  写锁（排他锁）:
    - 同一文件同一时间只能有一个工作流写入
    - 持有写锁时，其他工作流不能读或写
    - 写操作前必须检查是否有冲突

  锁的生命周期:
    - 申请: 工作流开始修改文件前
    - 持有: 修改过程中
    - 释放: 修改完成并验证后
    - 超时: 持有超过30分钟自动释放并告警

实现方式:
  - 通过 Git 分支隔离（推荐）
  - 通过 Todo 标注文件占用
  - 通过工作流元数据的 conflict_zones 声明
```

## 三、数据访问权限控制

### 3.1 目录权限矩阵

> **核心原则**: 每个工作流只能操作自己领域的目录，跨领域必须审批。

```yaml
权限矩阵:
  wf-fe-* (前端工作流):
    可读写: frontend/**
    可读取: backend/internal/model/**     # 读取数据模型定义
    可读取: backend/docs/**               # 读取API文档
    禁止写: backend/**、ai-service/**、yolo-service/**

  wf-be-* (后端工作流):
    可读写: backend/**
    可读取: frontend/src/api/**           # 了解前端调用方式
    禁止写: frontend/**、ai-service/**、yolo-service/**

  wf-ai-* (AI工作流):
    可读写: ai-service/**
    可读取: backend/internal/model/**     # 了解数据结构
    禁止写: frontend/**、backend/**、yolo-service/**

  wf-yolo-* (YOLO工作流):
    可读写: yolo-service/**
    可读取: backend/internal/model/**     # 了解数据结构
    禁止写: frontend/**、backend/**、ai-service/**

  wf-devops-* (运维工作流):
    可读写: deploy/**、scripts/**、.github/**、docker-compose*.yaml、Dockerfile*
    可读取: 所有目录
    禁止写: frontend/src/**、backend/internal/**、ai-service/app/**、yolo-service/app/**

  wf-cross-* (跨领域工作流):
    需要 project-lead 审批后才能操作
    审批时必须明确列出所有涉及目录
```

### 3.2 共享资源访问

```yaml
共享资源:
  配置文件:
    - .env、config.yaml 等全局配置
    - 规则: 任何工作流修改全局配置前必须通知所有相关方
    - 建议: 每个工作流使用独立的 .env.{workflow-id} 文件

  数据库:
    - DDL 操作（建表、改表）: 必须独占，通过数据库迁移文件排队
    - DML 操作（增删改查）: 按租户ID或工作流ID隔离数据
    - 规则: 测试数据必须带有工作流标识，方便清理

  类型定义:
    - 共享类型文件（如 API 接口定义）
    - 规则: 修改共享类型必须通过 code-review 技能审查
    - 建议: 新增类型优于修改已有类型

  路由注册:
    - backend/internal/router/router.go
    - 规则: 每个工作流在独立文件中注册路由，最后由 project-lead 统一合并
```

### 3.3 数据隔离策略

```yaml
数据隔离层级:

  L1 - 目录隔离（默认）:
    - 不同领域工作流在不同目录工作
    - 适用于大多数场景
    - 示例: 前端在 frontend/，后端在 backend/

  L2 - 分支隔离（推荐）:
    - 每个工作流在独立 Git 分支开发
    - 分支命名: feature/{workflow-id}
    - 完成后通过 PR 合并到主分支
    - 合并时由对应 Lead 审查冲突

  L3 - 数据隔离（特殊场景）:
    - 测试数据按工作流ID隔离
    - 数据库使用不同的 schema 或 database
    - 缓存使用不同的 key prefix
    - 示例: Redis key = wf-002:session:{id}

  L4 - 环境隔离（生产级）:
    - 使用 Docker Compose 的独立实例
    - 完全独立的数据库、缓存、服务
    - 仅用于重大架构变更的验证
```

## 四、执行优先级管理

### 4.1 优先级定义

| 等级 | 名称 | 触发条件 | 抢占能力 |
|------|------|----------|----------|
| P0 | 紧急 | 生产故障、安全漏洞 | 可抢占一切 |
| P1 | 高 | 核心功能、阻塞他人 | 可抢占 P2/P3 |
| P2 | 中 | 常规功能开发 | 可抢占 P3 |
| P3 | 低 | 优化、文档、重构 | 不抢占 |

### 4.2 优先级决策树

```
收到新任务
    │
    ├── 生产环境挂了？ ──── 是 ──→ P0（紧急）
    │
    ├── 安全漏洞？ ──────── 是 ──→ P0（紧急）
    │
    ├── 阻塞其他工作流？ ─── 是 ──→ P1（高）
    │
    ├── 核心功能需求？ ──── 是 ──→ P1（高）
    │
    ├── 常规功能需求？ ──── 是 ──→ P2（中）
    │
    ├── Bug修复？ ──────── 是 ──→ P2（中）
    │
    ├── 性能优化？ ──────── 是 ──→ P3（低）
    │
    └── 文档/重构？ ──────── 是 ──→ P3（低）
```

### 4.3 优先级调度规则

```yaml
调度规则:
  同级别工作流:
    - 先到先执行
    - 同一领域内串行执行
    - 不同领域可并行执行

  抢占规则:
    - P0 可以中断任何正在执行的工作流
    - P1 可以中断 P2 和 P3 的工作流
    - P2 可以中断 P3 的工作流
    - 被抢占的工作流必须保存进度并暂停

  暂停与恢复:
    暂停流程:
      1. 保存当前修改（Git stash 或 commit 到分支）
      2. 释放占用的端口和终端
      3. 记录暂停状态和进度
      4. 通知 project-lead

    恢复流程:
      1. 恢复修改（Git stash pop 或 checkout 分支）
      2. 重新申请端口和终端
      3. 从暂停点继续执行
      4. 通知 project-lead

  并行限制:
    同一目录: 最多1个工作流写入
    同一领域: 最多2个工作流并行（1写+1读）
    跨领域: 不限制并行数量
    全局: 最多5个工作流同时活跃
```

### 4.4 依赖管理

```yaml
依赖规则:
  声明依赖:
    每个工作流必须在元数据中声明依赖
    dependencies: [wf-be-user-api-002]

  依赖类型:
    hard: 必须等待前置工作流完成才能开始
    soft: 前置工作流完成后自动触发，但不阻塞

  依赖图:
    wf-be-user-api-002 (后端用户API)
        │
        ├── wf-fe-login-page-001 (前端登录页面) [hard]
        │
        └── wf-ai-detection-003 (AI检测) [soft]

  死锁检测:
    - 循环依赖 → 启动时检测并拒绝
    - 自依赖 → 启动时检测并拒绝
    - 依赖不存在 → 启动时检测并告警
```

## 五、冲突检测与解决机制

### 5.1 冲突类型

```yaml
冲突分类:

  文件冲突（最常见）:
    描述: 两个工作流同时修改同一文件
    危险等级: 🔴 高
    检测方式: 工作流元数据的 conflict_zones 对比

  端口冲突:
    描述: 两个工作流使用同一端口
    危险等级: 🔴 高
    检测方式: 端口分配表对比 + lsof 检查

  数据库冲突:
    描述: 两个工作流同时执行 DDL 操作
    危险等级: 🟡 中
    检测方式: 数据库迁移文件冲突检测

  API契约冲突:
    描述: 后端修改了API接口但前端不知道
    危险等级: 🟡 中
    检测方式: API文档版本对比

  配置冲突:
    描述: 两个工作流修改了同一配置文件
    危险等级: 🟡 中
    检测方式: Git diff 检测
```

### 5.2 冲突检测流程

```
新工作流启动
    │
    ▼
检查元数据 ──────────────────────────────────────
    │                                              │
    ├── 1. 对比 conflict_zones                     │
    │   与所有活跃工作流的 conflict_zones 取交集    │
    │   交集不为空 → 标记为潜在冲突                 │
    │                                              │
    ├── 2. 对比端口需求                            │
    │   检查端口分配表是否有重复                    │
    │   有重复 → 标记为端口冲突                     │
    │                                              │
    ├── 3. 对比目录范围                            │
    │   检查 scope.directories 是否有重叠           │
    │   有重叠且包含写操作 → 标记为目录冲突         │
    │                                              │
    ├── 4. 检查依赖图                              │
    │   检测循环依赖                                │
    │   检测依赖是否存在                            │
    │                                              │
    └── 5. 检查 Git 状态                           │
        检查目标文件是否有未提交的修改              │
        有未提交修改 → 标记为文件冲突               │
                                                   │
    ▼                                              │
生成冲突报告 ──────────────────────────────────────
    │                                              │
    ├── 无冲突 → ✅ 启动工作流                     │
    │                                              │
    └── 有冲突 → ⚠️ 进入冲突解决流程               │
```

### 5.3 冲突解决策略

```yaml
冲突解决优先级:

  策略一：等待（推荐）
    条件: 冲突工作流即将完成（<30分钟）
    做法: 等待对方完成后自动启动
    适用: 大多数文件冲突

  策略二：协商
    条件: 两个工作流都需要立即执行
    做法:
      1. 通知 project-lead
      2. 两个工作流的 owner 沟通
      3. 确定执行顺序或分工
      4. 记录协商结果
    适用: 端口冲突、复杂文件冲突

  策略三：隔离
    条件: 冲突可以通过数据隔离解决
    做法:
      1. 使用独立分支开发
      2. 使用独立端口
      3. 使用独立测试数据
      4. 最后合并时解决冲突
    适用: 同一领域内的并行开发

  策略四：抢占
    条件: 高优先级工作流必须立即执行
    做法:
      1. 通知被抢占工作流的 owner
      2. 被抢占方执行暂停流程
      3. 高优先级工作流立即启动
      4. 高优先级完成后恢复被抢占方
    适用: P0/P1 抢占 P2/P3

  策略五：合并
    条件: 两个工作流修改文件的不同部分
    做法:
      1. 各自在独立分支开发
      2. 都完成后由 Lead 执行合并
      3. 合并时解决冲突
      4. 运行测试验证
    适用: 大型功能并行开发
```

### 5.4 冲突解决检查清单

```yaml
冲突解决后必须检查:
  □ 两个工作流的修改是否都保留了
  □ 合并后的代码能否编译通过
  □ 合并后的测试能否全部通过
  □ 端口是否正确分配
  □ 数据库迁移是否按顺序执行
  □ API 契约是否对齐（前后端）
  □ 配置文件是否合并正确
  □ 没有遗留的临时代码或注释
```

## 六、并行执行限制条件

### 6.1 全局并行限制

```yaml
硬性限制（不可突破）:
  终端数量: 最多5个同时运行
  同一文件: 最多1个工作流写入
  同一端口: 最多1个工作流占用
  数据库DDL: 最多1个工作流执行

软性限制（可协商）:
  同一领域: 最多2个工作流并行
  全局活跃: 最多5个工作流
  每人并行任务: 最多3个
  待审查PR: 最多10个
```

### 6.2 安全并行条件

> **两个工作流可以安全并行的充分必要条件**：

```yaml
安全并行检查:
  必须全部满足:

    1. 目录不重叠:
       workflow_A.scope.directories ∩ workflow_B.scope.directories = ∅
       或者重叠目录中两个工作流都是只读

    2. 端口不冲突:
       workflow_A.resources.ports ∩ workflow_B.resources.ports = ∅

    3. 无依赖循环:
       workflow_A 不在 workflow_B 的依赖链中，反之亦然

    4. 数据库操作不冲突:
       两个工作流不同时对同一张表执行 DDL

    5. 共享资源已协调:
       如果都使用了共享类型/配置，变更已同步
```

### 6.3 并行执行矩阵

> 以下矩阵表示两个工作流是否可以并行执行。

```yaml
并行矩阵（✅ = 可并行，❌ = 不可并行，⚠️ = 需协调）:

              wf-fe-*  wf-be-*  wf-ai-*  wf-yolo-*  wf-devops-*
  wf-fe-*       ❌       ⚠️       ✅        ✅         ⚠️
  wf-be-*       ⚠️       ❌       ⚠️        ⚠️         ⚠️
  wf-ai-*       ✅       ⚠️       ❌        ⚠️         ✅
  wf-yolo-*     ✅       ⚠️       ⚠️        ❌         ✅
  wf-devops-*   ⚠️       ⚠️       ✅        ✅          ❌

说明:
  ❌ 同领域: 不可并行（资源冲突高风险）
  ⚠️ 跨领域但有关联: 需协调后并行
  ✅ 跨领域且无关联: 可自由并行
```

### 6.4 ⚠️ 需协调场景详解

```yaml
fe + be（前后端并行）:
  冲突点: API 契约
  协调方式:
    1. 先确定 API 接口文档（由 backend-lead 出）
    2. 前端基于接口文档开发 Mock
    3. 后端实现真实接口
    4. 最后联调替换 Mock
  关键: API 接口文档是前后端并行的"契约"

be + ai（后端 + AI 并行）:
  冲突点: 数据格式、接口协议
  协调方式:
    1. 先确定数据交互格式（JSON Schema）
    2. 后端和 AI 各自独立开发
    3. 通过统一的测试用例验证格式一致
  关键: 数据格式是协作的桥梁

fe + devops（前端 + 运维并行）:
  冲突点: 构建配置、环境变量
  协调方式:
    1. devops 不修改前端源码，只改配置
    2. 前端不修改部署配置
    3. 各自在独立文件中操作
  关键: 配置和源码分离

be + devops（后端 + 运维并行）:
  冲突点: Dockerfile、docker-compose、环境变量
  协调方式:
    1. devops 修改部署相关文件
    2. 后端修改业务代码
    3. 环境变量变更需要互相通知
  关键: 部署配置变更需通知
```

## 七、工作流生命周期管理

### 7.1 生命周期状态

```
  ┌──────────┐
  │ proposed │ ← 提出工作流
  └────┬─────┘
       ↓ 冲突检测通过
  ┌────▼─────┐
  │  active  │ ← 正在执行
  └────┬─────┘
       │
       ├──→ ┌──────────┐     ┌───────────┐
       │    │  paused  │────→│  active   │  ← 被抢占后恢复
       │    └──────────┘     └───────────┘
       │
       ├──→ ┌───────────┐
       │    │ blocked   │  ← 等待依赖
       │    └─────┬─────┘
       │          ↓ 依赖完成
       │    ┌─────▼─────┐
       │    │  active   │
       │    └───────────┘
       │
       └──→ ┌───────────┐
            │ completed │ ← 执行完成
            └───────────┘
```

### 7.2 启动检查清单

```yaml
工作流启动前必须完成:

  □ 命名符合规范（wf-{domain}-{feature}-{sequence}）
  □ 元数据已填写完整
  □ 冲突检测已执行且无阻塞性冲突
  □ 依赖工作流已完成（或有 soft 依赖说明）
  □ 端口已分配且无冲突
  □ 终端资源可用
  □ Git 分支已创建（feature/{workflow-id}）
  □ 对应 Lead 已审批（P0/P1 必须）
```

### 7.3 结束检查清单

```yaml
工作流结束前必须完成:

  □ 所有代码修改已提交到分支
  □ PR 已创建并指定 Reviewer
  □ 测试已通过
  □ 端口已释放
  □ 终端已清理
  □ 临时文件已删除
  □ 工作流状态已更新为 completed
  □ 通知依赖此工作流的其他工作流
```

## 八、紧急情况处理

### 8.1 工作流崩溃

```yaml
处理流程:
  1. 立即停止当前操作
  2. 评估影响范围（是否影响了其他工作流）
  3. 如有影响，立即通知相关方
  4. 释放所有占用资源
  5. 保存当前进度（Git commit）
  6. 记录崩溃原因和现场信息
  7. 修复后重新启动

崩溃恢复:
  - 优先恢复 P0/P1 工作流
  - 崩溃超过3次 → 升级到 project-lead
  - 无法恢复 → 启动 rollback 技能
```

### 8.2 资源耗尽

```yaml
处理流程:
  终端耗尽:
    - 检查是否有空闲终端可释放
    - 按优先级暂停最低的工作流
    - 通知被暂停工作流的 owner

  端口耗尽:
    - 检查是否有已停止但未释放的端口
    - 扩展端口池（使用备用范围）
    - 通知 project-lead

  并行上限:
    - 按优先级排队
    - P3 工作流自动让路
    - 通知排队中的工作流 owner
```

## 九、禁止事项

```yaml
🚫 绝对禁止:
  - 不做冲突检测就启动工作流
  - 强制占用已被其他工作流使用的端口
  - 修改不属于自己领域的文件（未经审批）
  - 同时对同一文件执行写操作
  - 跳过工作流元数据声明
  - 工作流结束后不释放资源
  - 直接修改 main 分支而不通过 PR
  - 无视抢占通知继续执行

⚠️ 强烈不建议:
  - 5个工作流以上同时活跃
  - 同一领域3个以上工作流并行
  - 长时间持有文件锁（>30分钟）
  - 不声明依赖就启动工作流
  - 忽略冲突检测警告
```

## 十、快速参考卡

### 启动工作流

```bash
# 1. 创建分支
git checkout -b feature/wf-fe-login-001

# 2. 检查端口占用
lsof -i :5100

# 3. 开始开发
pnpm dev --port 5100

# 4. 完成后提交
git add . && git commit -m "feat(wf-fe-login-001): 实现登录页面"
git push origin feature/wf-fe-login-001

# 5. 创建PR
gh pr create --title "feat: 登录页面" --body "工作流: wf-fe-login-001"
```

### 检查冲突

```bash
# 检查文件冲突
git diff main --name-only

# 检查端口占用
lsof -i :5100 -i :8200

# 检查活跃工作流
git branch -a | grep "feature/wf-"
```

### 紧急暂停

```bash
# 保存进度
git stash save "wf-fe-login-001 paused by P0 task"

# 释放端口（找到进程并终止）
lsof -i :5100 | grep LISTEN | awk '{print $2}' | xargs kill

# 切换到紧急任务
git checkout -b feature/wf-be-hotfix-006
```
