> **核心使命**: 让知识不再流失，让文档成为资产！📚

---

## 一、核心原则

```yaml
黄金法则:
  1️⃣ 代码即文档: 好的代码是最好的文档
  2️⃣ 文档同步: 代码变，文档必须变
  3️⃣ 简洁明了: 能用一句话说清，不用一段话
  4️⃣ 实用为主: 文档是为了用，不是为了写

文档优先级:
  1️⃣ API文档 (如何使用)
  2️⃣ 架构文档 (如何设计)
  3️⃣ 开发指南 (如何开发)
  4️⃣ 运维文档 (如何运维)
```

## 二、文档类型

### 2.1 API文档

**核心内容**：

```yaml
API文档要素:
  📝 端点描述
     - 请求方法 (GET/POST/PUT/DELETE)
     - 请求路径 (/api/users)
     - 功能说明 (获取用户列表)
  
  📥 请求参数
     - 参数名称
     - 参数类型
     - 是否必填
     - 参数说明
     - 示例值
  
  📤 响应格式
     - 成功响应结构
     - 失败响应结构
     - 状态码说明
     - 响应示例
  
  💡 使用示例
     - curl命令
     - 编程语言调用
     - 常见场景

文档模板:
  ## GET /api/users

  获取用户列表

  ### 请求参数
  | 参数 | 类型 | 必填 | 说明 |
  |------|------|------|------|
  | page | int | 否 | 页码，默认1 |
  | size | int | 否 | 每页数量，默认20 |

  ### 响应示例
  ```json
  {
    "code": 200,
    "data": {
      "users": [...],
      "total": 100
    }
  }
  ```

  ### 示例
  ```bash
  curl /api/users?page=1&size=20
  ```
```

## 三、文档编写规范

### 3.1 命名规范

```yaml
命名原则:
  ✅ 清晰: 一眼看出文档内容
  ✅ 一致: 统一命名风格
  ✅ 简洁: 不要太啰嗦
  
命名格式:
  ✅ 标题式
     README.md
     API.md
     DEPLOY.md
  
  ✅ 描述式
     用户模块文档.md
     订单流程说明.md
  
  ❌ 避免
     doc.txt
     新建文档.docx
     123.md
```

### 3.2 结构规范

```yaml
文档结构:
  📄 文件结构
     README.md (根目录必须有)
     ├── api/          # API文档
     ├── guides/       # 开发指南
     ├── architecture/ # 架构文档
     └── ops/          # 运维文档
  
  📝 单个文档结构
     1. 标题
     2. 概述
     3. 详细说明
     4. 示例
     5. 相关文档

目录规范:
  ✅ 按功能组织
     docs/
     ├── user/
     │   ├── README.md
     │   └── api.md
     └── order/
         ├── README.md
         └── api.md
  
  ❌ 不要按类型组织
     docs/
     ├── api/
     │   ├── user.md
     │   └── order.md
     └── guides/
         ├── user.md
         └── order.md
```

### 3.3 格式规范

```yaml
Markdown规范:
  ✅ 使用标题层级
     # 一级标题
     ## 二级标题
     ### 三级标题
  
  ✅ 使用列表
     - 无序列表
     1. 有序列表
  
  ✅ 使用代码块
     ```语言
     代码
     ```
  
  ✅ 使用表格
     | 列1 | 列2 |
     |------|------|
     | 内容 | 内容 |

  ❌ 不要
     - 长段落不分段
     - 代码和文字混在一起
     - 没有示例
```

### 3.4 注释规范

```yaml
代码注释原则:
  ✅ 解释为什么
     // 使用折扣需要VIP满3个月
     if (user.vipMonths >= 3) {
       applyDiscount()
     }
  
  ✅ 解释复杂逻辑
     // 二分查找: O(log n) 比线性查找快
     
  ✅ 标记TODO和FIXME
     // TODO: 优化性能
     // FIXME: 边界条件未处理

❌ 不要
  # 解释显而易见的代码
  // 增加计数器
  count++
  
  # 过时的注释
  // 曾经用于...
  
  # TODO不说明原因
  // TODO: 优化
```

## 六、文档质量

### 6.1 质量标准

```yaml
质量维度:
  ✅ 完整性
     - 覆盖所有功能
     - 包含示例
     - 解答常见问题
  
  ✅ 准确性
     - 内容正确
     - 示例可运行
     - 与代码一致
  
  ✅ 可读性
     - 结构清晰
     - 语言简洁
     - 格式规范
  
  ✅ 可维护性
     - 易于更新
     - 版本控制
     - 专人负责

评分标准:
  A: 完整、准确、可读、可维护
  B: 缺少某些内容
  C: 有错误或不准确
  D: 内容混乱，难以理解
```
