# Test Strategy - 测试策略大师

> **核心使命**: 用最少的测试覆盖最大的风险！🧪

---

## 一、核心原则

```yaml
黄金法则:
  1️⃣ 测试真实行为: 测试"做什么"而非"怎么做"
  2️⃣ 测试边界条件: Bug藏在边界，测试边界
  3️⃣ 测试应该独立: 每个测试独立运行
  4️⃣ 快速反馈: 测试应该快，能频繁运行

测试优先级:
  1️⃣ 核心业务逻辑
  2️⃣ 边界条件
  3️⃣ 错误处理
  4️⃣ 性能测试
  5️⃣ 安全测试
```

## 二、测试金字塔

### 2.1 金字塔结构

```yaml
        ┌─────────┐
        │   E2E   │ 少量、端到端、慢
        │  Tests  │ 验证完整流程
        └─────────┘
       ┌───────────┐
       │ Integration│ 中等数量、中等速度
       │  Tests   │ 验证模块协作
       └───────────┘
      ┌─────────────┐
      │   Unit     │ 大量、快速、隔离
      │  Tests     │ 测试最小单元
      └─────────────┘

理想比例:
  单元测试: 70%
  集成测试: 20%
  E2E测试: 10%

实际比例 (允许波动):
  单元测试: 60-80%
  集成测试: 15-25%
  E2E测试: 5-15%
```

### 2.2 各层测试特点

```yaml
单元测试 (Unit Tests)
  目标: 测试最小可测试单元
  范围: 函数、方法、类
  数量: 最多
  速度: 最快 (<10ms)
  隔离性: 完全隔离
  依赖: Mock所有依赖

集成测试 (Integration Tests)
  目标: 测试模块协作
  范围: 模块间接口
  数量: 中等
  速度: 中等 (10-100ms)
  隔离性: 部分隔离
  依赖: 只Mock外部系统

E2E测试 (End-to-End Tests)
  目标: 验证完整业务流程
  范围: 整个系统
  数量: 最少
  速度: 最慢 (100ms-10s)
  隔离性: 无隔离
  依赖: 真实环境
```

## 三、测试策略

### 3.1 TDD开发流程

```yaml
TDD三定律:
  1️⃣ 在写代码前，先写一个失败的测试
  2️⃣ 只写刚好让测试通过的最少代码
  3️⃣ 只写能让测试通过的生产代码

TDD流程:
  Red (红):
    ┌──────────────────────┐
    │ 1. 写一个失败的测试   │
    │ 2. 运行测试，确认失败  │
    │ 3. 明确要实现什么     │
    └──────────────────────┘
  
  Green (绿):
    ┌──────────────────────┐
    │ 4. 写最简单的代码    │
    │ 5. 运行测试，确认通过 │
    │ 6. 不考虑代码质量     │
    └──────────────────────┘
  
  Refactor (重构):
    ┌──────────────────────┐
    │ 7. 重构代码           │
    │ 8. 测试保持通过       │
    │ 9. 改善代码质量       │
    └──────────────────────┘
```

### 3.2 BDD行为驱动

```yaml
BDD结构:
  Feature: 功能名称
    Scenario: 场景描述
      Given: 前提条件
      When: 执行动作
      Then: 预期结果

示例:
  Feature: 用户登录
    Scenario: 正确密码登录成功
      Given 用户在登录页面
      When 输入正确用户名和密码
      Then 登录成功，跳转到首页

    Scenario: 错误密码登录失败
      Given 用户在登录页面
      When 输入错误密码
      Then 登录失败，显示错误提示
```

### 3.3 测试用例设计

```yaml
等价类划分:
  输入 → 分组 → 每个组选代表测试
  
  示例:
    年龄输入 (0-150)
    - 无效: 负数, >150
    - 有效: 0-150 之间

边界值分析:
  测试边界值本身及其相邻值
  
  示例:
    边界: 0, 1, 149, 150
    相邻: -1, 1, 149, 151

因果图法:
  分析输入组合和输出关系
  
  组合测试:
    - 所有单条件组合
    - 所有条件组合
    - 重要组合
```

## 四、测试覆盖策略

### 4.1 覆盖维度

```yaml
覆盖类型:
  ✅ 语句覆盖
     - 每个语句至少执行一次
     - 最基础的覆盖
  
  ✅ 判定覆盖
     - 每个分支至少执行一次
     - if/else都测试
  
  ✅ 条件覆盖
     - 每个条件的每个值都测试
     - 复合条件的每个部分
  
  ✅ 路径覆盖
     - 每个可能的路径都测试
     - 最完整的覆盖

覆盖目标:
  核心业务: 90%+
  普通业务: 80%+
  辅助代码: 60%+
```

### 4.2 测试用例设计原则

```yaml
AAA模式:
  Arrange: 准备测试数据
  Act: 执行被测操作
  Assert: 验证结果

示例:
  test('计算总价', () => {
    // Arrange
    const items = [
      { price: 100, quantity: 2 },
      { price: 50, quantity: 1 }
    ]
    
    // Act
    const total = calculateTotal(items)
    
    // Assert
    expect(total).toBe(250)
  })

好的测试特征:
  ✅ 描述清晰: 测试名称说明测试什么
  ✅ 独立: 测试之间不依赖
  ✅ 可重复: 每次运行结果一致
  ✅ 快速: 运行时间短
  ✅ 自我验证: 测试自己判断通过失败
```

### 4.3 避免的测试

```yaml
❌ 不要测试:
  - 内部实现细节
  - 私有方法
  - 无关细节
  - 无聊的测试

❌ 测试的反模式:
  - 测试实现而非行为
  - 测试没有断言
  - 测试过于复杂
  - 测试重复
  - 测试脆弱

⚠️ 警告信号:
  - 测试名称难以理解
  - 测试包含大量mock
  - 测试与实现耦合
  - 测试经常失败
```

## 五、测试质量保障

### 5.1 测试审查清单

```yaml
审查维度:
  ✅ 可读性
     - 测试名称清晰
     - 包含Arrange-Act-Assert
     - 有必要的注释
  
  ✅ 可维护性
     - 测试独立
     - 易于修改
     - 清晰的fixture
  
  ✅ 可靠性
     - 稳定不 flaky
     - 无外部依赖
     - 清理测试数据
  
  ✅ 覆盖率
     - 核心逻辑覆盖
     - 边界条件覆盖
     - 错误处理覆盖
```

### 5.2 测试质量指标

```yaml
量化指标:
  📊 代码覆盖率
     - 行覆盖率: 80%+
     - 分支覆盖率: 75%+
     - 函数覆盖率: 90%+
  
  📊 测试数量
     - 单元测试: 业务代码 × 2-3
     - 集成测试: 关键路径 × 5
     - E2E测试: 核心流程 × 3
  
  📊 测试质量
     - Flaky测试率: < 1%
     - 测试运行时间: < 5分钟
     - 测试通过率: > 95%

质量报告模板:
```

## 六、各语言测试实践

### 6.1 前端测试

```yaml
单元测试:
  框架: Jest, Vitest
  目标: 组件逻辑、工具函数
  覆盖率: 80%+
  
  示例:
    test('计算总价正确', () => {
      expect(calculateTotal(items)).toBe(250)
    })

组件测试:
  框架: React Testing Library, Vue Test Utils
  目标: 组件行为
  原则: 测试行为，不测试实现

E2E测试:
  框架: Playwright, Cypress
  目标: 完整用户流程
  场景: 核心业务流程
```

### 6.2 Go测试

```yaml
测试框架: go test
Mock: testify/mock, gomock

单元测试:
  示例:
    func TestCalculateTotal(t *testing.T) {
      items := []Item{
        {Price: 100, Quantity: 2},
      }
      total := CalculateTotal(items)
      if total != 200 {
        t.Errorf("expected 200, got %d", total)
      }
    }

基准测试:
    func BenchmarkCalculateTotal(b *testing.B) {
      for i := 0; i < b.N; i++ {
        CalculateTotal(items)
      }
    }
```

### 6.3 Python测试

```yaml
测试框架: pytest, unittest
Mock: pytest-mock, unittest.mock

示例:
  def test_calculate_total():
      items = [
          {'price': 100, 'quantity': 2},
      ]
      total = calculate_total(items)
      assert total == 200

Fixtures:
  @pytest.fixture
  def sample_items():
      return [
          {'price': 100, 'quantity': 2},
      ]
```

## 十、总结 📝

**Test Strategy 核心职责**：

1. 🧪 **测试金字塔** - 合理分配测试资源
2. ✅ **TDD实践** - 测试驱动开发
3. 📊 **覆盖率保障** - 确保关键代码被测
4. 🔍 **质量审查** - 保证测试质量
5. 🚀 **持续测试** - 集成到CI/CD

**记住**：
- 测试金字塔比测试数量更重要
- 测试真实行为，不测试实现
- 好的测试应该快、独立、可重复
- 覆盖率是手段，不是目的
