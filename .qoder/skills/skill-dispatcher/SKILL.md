# Skill Dispatcher - 智能技能调度器

> **核心使命**: 让每个任务都能找到最合适的技能，让每个技能都能在正确的时机被调用

---

## 一、核心原则

```yaml
优先级法则:
  1️⃣ 优先调用技能: 遇到任何开发任务，先检查是否有相关技能
  2️⃣ 精准匹配: 根据任务类型智能匹配最合适的技能
  3️⃣ 避免重复: 一个任务只调用最相关的一个技能
  4️⃣ 失败回退: 如果没有匹配的技能，自己处理

黄金法则:
  - 有技能，必调用
  - 先分析，后调度
  - 匹配度，看描述
  - 不确定，问用户
```

### 3.2 匹配算法

```python
# 伪代码：技能匹配算法
def match_skill(user_request):
    # 1. 提取关键词
    keywords = extract_keywords(user_request)
    
    # 2. 计算匹配分数
    scores = {}
    for skill in skill_library:
        score = calculate_match_score(keywords, skill.keywords)
        scores[skill.name] = score
    
    # 3. 选择最高分技能
    best_skill = max(scores, key=scores.get)
    
    # 4. 阈值判断
    if scores[best_skill] > THRESHOLD:
        return best_skill
    else:
        return None  # 自己处理
```

## 五、特殊场景处理

### 5.1 多技能协同

```yaml
场景: 任务涉及多个技术栈
示例: "开发一个带AI检测的视频监控页面"

处理流程:
  1. 识别主要任务: 前端页面开发
  2. 识别次要任务: AI检测集成
  3. 调用策略:
     - 主要技能: frontend-dev (负责页面)
     - 协同技能: ai-lead (提供AI接口)
  
决策原则:
  - 优先调用主要任务对应的技能
  - 主要技能可以调用其他技能协助
  - 复杂任务直接交给project-lead
```

### 5.2 技能不可用

```yaml
场景: 匹配的技能不存在或不可用
示例: 匹配到frontend-dev但技能库中没有

处理方式:
  1. 降级处理: 尝试匹配相似技能
     - frontend-dev → frontend-lead
  2. 再次降级: 使用通用技能
     - frontend-lead → project-dev
  3. 最终降级: 自己处理
     - 使用global-dev规范自己实现
```

### 5.3 用户明确指定

```yaml
场景: 用户明确指定要使用的技能
示例: "使用backend-dev开发这个API"

处理方式:
  - 直接调用用户指定的技能
  - 不进行自动匹配
  - 尊重用户选择
```

## 七、最佳实践

### 7.1 调用时机

```yaml
✅ 必须调用:
  - 开始任何开发任务前
  - 用户明确要求使用某个技能
  - 任务涉及特定技术栈
  - 需要专业领域知识

⚠️ 可选调用:
  - 简单的文档查询
  - 纯信息性问题
  - 已经在某个技能执行过程中

❌ 不应调用:
  - 用户明确表示不需要
  - 任务与开发无关
  - 正在执行紧急修复
```

### 7.2 调用顺序

```
优先级从高到低:
  1. 用户明确指定的技能
  2. 匹配度最高的专业技能
  3. 相似领域的通用技能
  4. 自己处理
```

### 7.3 错误处理

```yaml
技能调用失败:
  - 记录失败原因
  - 尝试降级方案
  - 通知用户情况
  - 提供替代方案

技能不存在:
  - 检查技能名称拼写
  - 查看技能库列表
  - 使用相似技能替代
```

## 八、禁止事项

```yaml
❌ 绝对禁止:
  - 跳过技能匹配直接自己处理（有匹配技能时）
  - 同时调用多个技能处理同一任务
  - 忽略用户明确指定的技能
  - 在技能执行过程中调用其他技能（除非必要）
  - 调用与任务无关的技能

⚠️ 需要特别注意:
  - 技能调用的时机（任务开始前）
  - 技能匹配的准确性（避免误判）
  - 用户的明确意图（优先满足）
  - 技能执行的上下文（传递完整信息）
```
