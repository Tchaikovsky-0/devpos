## 📌 规范概述

本规范旨在为巡检宝项目的调试工作提供**系统化、科学化的方法**。核心目标是让Bug无处遁形，用最小的代价找到问题根源。

---

## 🎯 核心原则

### 1. 科学调试原则

```yaml
调试顺序:
  1️⃣ 复现问题 → 找到最小复现步骤
  2️⃣ 收集信息 → 错误日志、环境信息、数据
  3️⃣ 分析假设 → 列出所有可能的原因
  4️⃣ 验证排除 → 逐一验证，排除不可能
  5️⃣ 定位根因 → 找到真正的问题
  6️⃣ 修复验证 → 确认修复有效

核心法则:
  ✅ 复现为王: 找不到复现步骤就别想修好
  ✅ 科学方法: 假设-验证-结论
  ✅ 分而治之: 大问题拆成小问题
  ✅ 最小案例: 找到能复现问题的最小代码
```

---

## 📋 调试规则

### R1: 问题复现规则

**规则描述**：所有Bug必须先复现，才能修复。

**复现标准**：

```yaml
必须记录的信息:
  ✅ 问题现象
  ✅ 复现步骤
  ✅ 环境信息
  ✅ 复现率

复现率分级:
  - 100%必现: 容易定位
  - 偶发: 需要更多日志
  - 无法复现: 需要更多线索
```

---

### R2: 根因分析规则

**规则描述**：必须找到真正的根因，不能只修复表面现象。

**分析方法**：

```yaml
分层分析法:
  层级1: 用户界面层
  层级2: 请求层
  层级3: 业务逻辑层
  层级4: 数据层
  层级5: 系统层

常用技术:
  ✅ 二分查找
  ✅ 追溯分析
  ✅ 日志追踪
  ✅ 断点调试
```

---

### R3: 修复验证规则

**规则描述**：修复后必须验证，防止引入新问题。

**验证清单**：

```yaml
验证内容:
  ✅ 原问题已修复
  ✅ 没有引入新问题
  ✅ 边界条件正常
  ✅ 性能没有下降
  ✅ 相关功能正常

必须执行:
  ✅ 运行所有测试
  ✅ 回归测试
  ✅ 性能测试
```

---

## 🚫 禁止规则

### R4: 调试禁止规则

```yaml
🚫 绝对禁止:
  - 没有复现就修复Bug
  - 瞎猜问题原因
  - 边改代码边调试
  - 修复后不测试就上线
  - 忽略错误日志
  - 修复一个Bug引入另一个Bug

⚠️ 强烈不推荐:
  - 大范围修改
  - 同时修改多个地方
  - 不写测试的修复
```
