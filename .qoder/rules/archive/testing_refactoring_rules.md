# 第一部分：测试规范

## 🎯 测试核心原则

### 测试金字塔

```yaml
        ┌─────────┐
        │   E2E   │ 少量、慢
        │  Tests  │ 验证完整流程
        └─────────┘
       ┌───────────┐
       │ Integration│ 中等数量
       │  Tests   │ 验证模块协作
       └───────────┘
      ┌─────────────┐
      │   Unit     │ 大量、快速
      │  Tests     │ 测试最小单元
      └─────────────┘

理想比例:
  单元测试: 70%
  集成测试: 20%
  E2E测试: 10%
```

### TDD流程

```yaml
Red: 先写失败的测试
Green: 最快速度让测试通过
Refactor: 重构代码

原则:
  ✅ 测试真实行为
  ✅ 测试边界条件
  ✅ 测试错误处理
```

## 📋 测试规则

### 覆盖标准

```yaml
覆盖目标:
  核心业务逻辑: 90%+
  普通业务逻辑: 80%+
  辅助代码: 60%+

维度:
  ✅ 语句覆盖
  ✅ 分支覆盖
  ✅ 函数覆盖
  ✅ 路径覆盖
```

### AAA模式

```yaml
test('计算总价', () => {
  // Arrange: 准备测试数据
  const items = [{ price: 100, quantity: 2 }]
  
  // Act: 执行操作
  const total = calculateTotal(items)
  
  // Assert: 验证结果
  expect(total).toBe(200)
})
```

### 测试禁止规则

```yaml
🚫 绝对禁止:
  - 测试实现细节
  - 测试没有断言
  - 测试相互依赖
  - Flaky测试
  - 测试覆盖不足的核心逻辑
```

# 第二部分：重构规范

## 🎯 重构核心原则

### 重构前提

```yaml
重构必须满足:
  ✅ 有测试覆盖
  ✅ 功能正常工作
  ✅ 理解代码意图
  ✅ 制定重构计划

重构禁忌:
  ❌ 没有测试就重构
  ❌ 边重构边改功能
  ❌ 一次性大改
  ❌ 不理解代码就重构
```

### 安全重构三步法

```yaml
1️⃣ 添加新代码
2️⃣ 运行测试
3️⃣ 删除旧代码
```

### 童子军规则

```yaml
→ 每次离开时，代码比来时更干净
```

## 📋 重构规则

### 坏味道识别

```yaml
代码层面:
  - 重复代码 (>3次)
  - 过长函数 (>50行)
  - 过大类 (>300行)
  - 参数过多 (>4个)

设计层面:
  - 霰弹式修改
  - 特性依恋
  - 数据泥团
```

### 重构验证清单

```yaml
功能验证:
  ✅ 所有测试通过
  ✅ 原有功能正常

代码验证:
  ✅ 代码更简洁
  ✅ 命名更清晰
  ✅ 重复代码减少

性能验证:
  ✅ 性能没有下降
  ✅ 响应时间正常
```

### 重构禁止规则

```yaml
🚫 绝对禁止:
  - 没有测试就重构
  - 重构时改变功能
  - 一次性大范围重构
  - 不理解代码就重构
  - 重构后不运行测试
```
