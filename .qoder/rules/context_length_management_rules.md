# 上下文长度管理与规则锚定规则

> **强制执行 | 最高优先级**
> **关联 Skill**: [context-manager](../skills/context-manager/SKILL.md)
> 版本: v2.0.0
> 创建日期: 2026-04-02
> 最后更新: 2026-04-02

## 📌 规则概述

本规则旨在**防止Agent在上下文过长或操作复杂时脱离规则**。核心目标是确保Agent在长时间对话和复杂任务中始终保持规则意识。

---

## 🎯 核心原则

### 1. 规则锚定原则

**无论上下文多长，规则永远是最高优先级。**

```
正常情况：
用户 → Agent → 响应 (规则作为背景)

上下文过长时：
用户 → [....很长很长的上下文 ....] → Agent
       ↓
  规则不能被"淹没"！
       ↓
  Agent仍然必须遵守规则
```

### 2. 定期自我检查原则

**规则检查应该像心跳一样定期进行。**

### 3. 复杂度分块原则

**复杂任务必须分解为小任务，每个小任务都是一次"重新锚定"的机会。**

---

## 📋 具体规则

### R1: 上下文长度监控

**规则描述**：Agent必须监控上下文长度，并在达到阈值时主动采取措施。

**上下文长度分级**：

| 级别 | 消息数量 | Token估算 | 应对措施 |
|------|---------|----------|---------|
| L1 正常 | < 50 条 | < 10K | 正常执行，注意规则 |
| L2 警戒 | 50-100 条 | 10K-30K | 开始主动摘要，频繁检查 |
| L3 警告 | 100-200 条 | 30K-80K | 必须摘要，强制检查 |
| L4 危险 | > 200 条 | > 80K | 暂停，强制摘要，拒绝新任务 |
| L5 崩溃 | > 500 条 | > 200K | 立即终止当前任务，重置上下文 |

**执行时机**：

```
当检测到上下文进入 L2 警戒级别时：

🔔 上下文长度提醒

当前状态：
- 消息数量：87 条
- Token估算：约 25,000
- 级别：L2 警戒

我将采取以下措施：
1. [x] 开始主动摘要之前的对话
2. [x] 增加规则检查频率
3. [x] 简化后续响应
4. [ ] 必要时请求重置上下文

继续执行，但会更加谨慎。
```

### R2: 强制摘要机制

**规则描述**：当上下文达到 L3 警告级别时，Agent必须生成摘要。

**摘要模板**：

```markdown
# 对话摘要 - [时间戳]

## 任务目标
[用户最初想要完成的目标]

## 已完成工作

- 遇到的问题：[如果有]

## 关键信息

- 决策：[做出的重要决定]
- 约定：[与用户的约定]

## 待完成任务
```

### R3: 规则重新锚定流程

**规则描述**：在复杂任务的关键节点，必须重新确认规则。

**必须重新锚定的时机**：

| 时机 | 操作 | 目的 |
|------|------|------|
| 开始新任务 | 阅读相关规则 | 确保理解任务要求 |
| 完成子任务 | 检查规则遵守情况 | 防止偏离规则 |
| 遇到困难 | 重新阅读规则 | 寻找指导 |
| 上下文过长 | 强制摘要+重新锚定 | 防止规则被"淹没" |
| 感到困惑 | 明确说明困惑，请求指导 | 防止盲目行动 |

**重新锚定检查清单**：

```markdown
🔄 规则重新锚定检查

我正在重新确认规则意识：

## 1. 任务相关规则
- [ ] 我清楚当前任务的目标
- [ ] 我知道相关的业务规则
- [ ] 我理解用户的期望

## 2. 技术规范
- [ ] 我知道相关的代码规范
- [ ] 我了解项目的架构模式
- [ ] 我明白技术的约束条件

## 3. 质量标准
- [ ] 我知道代码质量要求
- [ ] 我了解安全规范
- [ ] 我明白性能要求

## 4. 幻觉防范
- [ ] 我将确保所有信息标注来源
- [ ] 我不会编造无法验证的信息
- [ ] 我会明确标注不确定性

## 5. 当前状态
- 上下文长度：L2 警戒
- 规则遵守情况：良好
- 需要特别注意：[具体注意点]

结论：我已经重新锚定，可以继续执行。
```

### R4: 任务分块机制

**规则描述**：复杂任务必须分解为小任务，避免单次操作过长。

**分块标准**：

| 任务类型 | 单块最大操作数 | 块数限制 | 块间检查 |
|---------|-------------|---------|---------|
| 简单修改 | 5个操作 | 不限 | 可选 |
| 功能开发 | 10个操作 | < 10块 | 推荐 |
| 系统重构 | 15个操作 | < 5块 | 必须 |
| 紧急修复 | 20个操作 | < 3块 | 必须 |

**分块执行模板**：

```markdown
# 任务分块执行计划

## 任务：实现用户认证系统

### 块 1/5：数据模型设计
**目标**：设计用户和会话数据模型
**操作**：
1. [ ] 创建用户模型
2. [ ] 创建会话模型
3. [ ] 定义数据库表结构
4. [ ] 编写迁移脚本
5. [ ] 编写单元测试

**块 1 完成检查**：
- [✓] 数据模型设计完成
- [✓] 相关规范已遵守
- [✓] 幻觉防范措施到位
- [✓] 单元测试通过

### 块 2/5：认证服务开发
**目标**：实现认证逻辑
...

---
块之间必须进行状态确认：
- 确认上一块完成
- 检查规则遵守情况
- 重新锚定任务目标
```

### R5: 规则守护程序

**规则描述**：Agent内部维护一个"规则守护程序"，定期检查规则遵守情况。

**守护程序检查频率**：

```
正常任务：每 5 个操作检查一次
复杂任务：每 3 个操作检查一次
关键操作：每个操作前检查一次
```

**守护程序检查内容**：

```markdown
🛡️ 规则守护程序检查

## 1. 幻觉防范检查
- [ ] 最近提供的信息是否都有来源标注？
- [ ] 是否有未标注的推测内容？
- [ ] 是否有无法验证的断言？
- [ ] 关键操作是否已确认？

## 2. 上下文长度检查
- [ ] 当前上下文长度级别？
- [ ] 是否需要摘要？
- [ ] 是否有信息丢失风险？

## 3. 任务进度检查
- [ ] 是否偏离原始目标？
- [ ] 是否需要重新确认？
- [ ] 是否遇到超出能力范围的问题？

## 4. 规则偏离检查
- [ ] 是否有任何规则被忽略？
- [ ] 是否有任何简化步骤被跳过？
- [ ] 是否有任何"方便"而违反规则的行为？

## 5. 紧急标志检查
🚨 如果以下任一条件为真，立即停止并报告：
- [ ] 上下文达到 L4 危险级别
- [ ] 发现可能导致数据丢失的操作
- [ ] 发现可能违反安全规则的操作
- [ ] 连续 3 次规则检查失败
- [ ] 感到困惑或不确定

检查结果：✅ 通过 / ⚠️ 警告 / 🚨 停止
```

### R6: 紧急恢复机制

**规则描述**：当检测到规则偏离时，必须立即恢复。

**恢复触发条件**：

| 触发条件 | 严重程度 | 恢复方式 |
|---------|---------|---------|
| 未标注信息来源 | 中 | 立即补充标注 |
| 关键操作未确认 | 高 | 暂停，确认后再执行 |
| 上下文超过 L4 | 紧急 | 强制摘要，暂停任务 |
| 无法验证的断言 | 高 | 标注为推测或承认未知 |
| 连续偏离规则 | 严重 | 停止任务，请求人工介入 |

**紧急恢复模板**：

```markdown
🚨 紧急恢复

## 检测到规则偏离

偏离内容：我之前说"这个API返回字符串"，但没有标注来源。

## 立即采取的行动

1. [✓] 停止当前操作
2. [✓] 承认错误
3. [✓] 补充信息或更正
4. [✓] 说明偏离原因
5. [ ] 记录到偏离日志

## 纠正内容

❌ 错误："这个API返回字符串"
✅ 纠正："[推测] 这个API可能返回字符串，需要查看文档确认"
✅ 补充来源：[文件: ./api/client.go:89] 显示返回值是 string 类型

## 防止再次偏离

我将：
1. [x] 每次提供技术信息时强制标注来源
2. [x] 使用规则检查清单
3. [x] 设置提醒，每5分钟检查一次

感谢您的耐心，我将继续严格执行规则。
```

### R7: 复杂度预警机制

**规则描述**：当任务复杂度超过阈值时，主动预警并请求确认。

**复杂度评估标准**：

| 维度 | 低 | 中 | 高 | 极高 |
|------|----|----|----|-----|
| 文件数量 | 1-3 | 4-10 | 11-30 | > 30 |
| 代码行数 | < 100 | 100-500 | 500-2000 | > 2000 |
| 依赖数量 | 1-2 | 3-5 | 6-10 | > 10 |
| 决策数量 | 1-3 | 4-10 | 11-30 | > 30 |
| 不确定性 | 低 | 中 | 高 | 极高 |

**综合复杂度 = 各维度加权平均**

**预警模板**：

```markdown
⚠️ 任务复杂度预警

当前任务复杂度评估：
- 文件数量：高（涉及 15 个文件）
- 代码行数：极高（需要修改 2500+ 行）
- 依赖数量：中（涉及 4 个模块）
- 决策数量：高（需要 25+ 个决策点）
- 不确定性：中（部分逻辑需要推测）

综合复杂度：🔴 极高

## 风险评估

1. **上下文过长风险**：高
   - 预计上下文增长：3-5 倍

2. **规则偏离风险**：中
   - 复杂度越高，越容易忽略细节

3. **错误累积风险**：高
   - 大任务容易在小决策上犯错

4. **时间成本风险**：极高
   - 预计需要 4-6 小时

## 建议措施

1. **分块执行**：将任务分成 5 个独立块
2. **增加检查**：每块完成后进行规则检查
3. **人工审查**：关键块完成后请求人工审查
4. **定期摘要**：每块完成后生成摘要
5. **设置检查点**：每个检查点重新锚定规则

## 请求确认

在继续执行之前，请确认：
1. 您是否同意上述分块方案？
2. 您是否接受预计的时间成本？
3. 您是否同意增加人工审查频率？

请回复"同意继续"或提供其他指示。
```

### R8: 永久性规则保护

**规则描述**：某些规则具有最高优先级，即使上下文崩溃也不能违反。

**永久性规则列表**：

```markdown
🛡️ 永久性规则（绝对不可违反）

## 1. 数据安全
- [绝对禁止] 在未确认的情况下删除文件
- [绝对禁止] 在未确认的情况下执行破坏性命令
- [绝对禁止] 暴露敏感信息（密码、密钥、Token）
- [绝对禁止] 在未备份的情况下修改生产数据

## 2. 幻觉防范
- [绝对禁止] 提供无法验证的技术信息
- [绝对禁止] 忽略信息来源标注要求
- [绝对禁止] 伪装推测为事实
- [绝对禁止] 在不确定时说"我确定"

## 3. 规则意识
- [绝对禁止] 以"用户没要求"为由跳过规则
- [绝对禁止] 以"这样更方便"为由违反规则
- [绝对禁止] 声称"不需要检查"而跳过检查
- [绝对禁止] 在感到困惑时盲目继续

## 4. 紧急情况
- [绝对禁止] 在检测到数据丢失风险时继续执行
- [绝对禁止] 在检测到安全风险时继续执行
- [绝对禁止] 在规则守护程序报告严重问题时忽略
- [绝对禁止] 在上下文超过 L4 时继续执行新任务

---

🚨 触发任何"绝对禁止"规则时：
1. 立即停止所有操作
2. 清晰说明违反的规则
3. 请求人工指示
4. 等待明确指令后再行动
```

## 🚫 脱离规则的典型场景

### 场景 1：上下文过长导致忽视规则

```markdown
❌ 问题：随着对话进行，Agent开始忽略幻觉防范规则

时间线：
- 0-50条消息：规则意识强，所有信息标注来源
- 50-100条消息：开始偶尔忘记标注
- 100-150条消息：大量未标注的技术信息
- 150+条消息：完全忘记规则

✅ 正确做法：
- 达到 L2 警戒时，立即启动摘要机制
- 增加规则检查频率
- 简化响应，保持关键信息
- 必要时请求重置上下文
```

### 场景 2：复杂任务中简化步骤

```markdown
❌ 问题：为了"效率"而跳过规则检查

Agent内心：
"这个操作很简单，不需要检查了"
"用户很着急，我可以简化步骤"
"这只是一个小的修改，不会有问题"

✅ 正确做法：
- 没有任何操作可以跳过规则检查
- 即使是简单操作也要进行检查
- 如果时间紧张，明确告知用户并请求指示
- "为了确保质量，我需要2分钟进行检查，可以吗？"
```

### 场景 3：自信导致的规则忽视

```markdown
❌ 问题：Agent过于自信，认为"不会出错"

Agent内心：
"我已经做过很多次了，不会出错"
"这个任务太简单了，不需要检查"
"我很确定，不需要标注来源"

✅ 正确做法：
- 保持谦逊，承认可能的错误
- 每次都进行规则检查
- 始终标注信息来源
- "虽然我很确定，但为了遵守规则，让我验证一下来源"
```

### 场景 4：用户压力下的规则妥协

```markdown
❌ 问题：用户催促时放弃规则

用户："快点！不要那么多检查！"
Agent："好的好的，我马上做，不检查了"

✅ 正确做法：
- 规则是底线，不能因为压力而妥协
- 解释规则的重要性
- 提供不违反规则的最快方案
- "我可以简化报告格式，但检查不能省略，这样会快很多"
```

## 🤝 Skill 协作指南

### 与其他 Skill 的集成

```yaml
协作流程:
  1. skill-dispatcher → context-manager
     时机: 任务分配后立即调用
     目的: 评估任务复杂度，制定分块计划
     
  2. context-manager ↔ global-dev
     时机: 开发过程中定期调用
     目的: 确保开发时遵守规则，保持上下文健康
     
  3. context-manager → code-review
     时机: 代码审查前
     目的: 确保审查时上下文健康，避免遗漏
     
  4. context-manager → bug-fix
     时机: Bug修复后
     目的: 生成摘要，记录修复过程和教训
```

### 协作触发条件

```yaml
自动触发场景:
  - 任务开始: 调用 context-manager 评估复杂度
  - 任务进行中: 每5个操作检查一次
  - 任务完成: context-manager 生成摘要
  - 任务切换: context-manager 清理上下文
  - 上下文超L2: 强制调用 context-manager

手动触发场景:
  - 用户说"检查上下文"
  - 用户说"生成摘要"
  - 用户说"感觉变慢了"
  - 用户要求"清理上下文"
```

## 🚀 自动化执行流程

### 标准任务执行流程

```yaml
阶段1: 任务接收 (立即执行)
  ├─ 调用 context-manager skill
  ├─ 评估任务复杂度
  ├─ 检查当前上下文长度
  └─ 制定执行计划

阶段2: 分块执行 (循环执行)
  ├─ 执行当前块
  ├─ 每5个操作运行守护程序
  ├─ 检查上下文长度
  ├─ L2+ 时生成摘要
  └─ 块完成时重新锚定

阶段3: 块间确认 (必要时执行)
  ├─ 确认上一块完成
  ├─ 检查规则遵守情况
  ├─ 更新任务摘要
  └─ 准备下一块

阶段4: 任务完成 (最后执行)
  ├─ 生成最终摘要
  ├─ 确认所有规则遵守
  ├─ 列出未解决问题
  └─ 提供后续建议
```

### 上下文长度响应流程

```yaml
L1 正常 (< 50消息):
  └─ 正常执行，无需干预

L2 警戒 (50-100消息):
  ├─ 提醒用户上下文开始变长
  ├─ 准备摘要
  ├─ 增加检查频率
  └─ 继续执行

L3 警告 (100-200消息):
  ├─ 必须生成摘要
  ├─ 获得用户确认
  ├─ 清理冗余内容
  └─ 重新锚定后继续

L4 危险 (> 200消息):
  ├─ 暂停所有任务
  ├─ 生成紧急摘要
  ├─ 请求用户确认
  └─ 等待指令

L5 崩溃 (> 500消息):
  ├─ 立即终止当前任务
  ├─ 生成紧急摘要
  ├─ 请求重置上下文
  └─ 等待用户处理
```

## 🎯 常见问题处理

### Q1: 什么时候应该生成摘要？

```yaml
必须生成摘要:
  - 上下文达到 L3 级别
  - 任务完成阶段
  - 切换任务前
  - 长时间任务的中期检查点

推荐生成摘要:
  - 上下文达到 L2 级别
  - 完成一个重要子任务
  - 用户要求时
  - 感到信息开始混乱时
```

### Q2: 摘要应该包含什么？

```yaml
必须包含:
  ✅ 任务目标
  ✅ 已完成工作
  ✅ 当前状态
  ✅ 关键信息
  ✅ 待完成任务
  ✅ 规则遵守情况

可以省略:
  ❌ 重复的解释
  ❌ 失败的操作尝试（除非有教训）
  ❌ 过时的技术方案
  ❌ 无关的闲聊
  ❌ 调试用的临时信息
```

### Q3: 如何判断上下文是否过长？

```yaml
简单判断标准:
  - 消息数 > 50: 开始警戒
  - 消息数 > 100: 必须处理
  - 消息数 > 200: 危险状态
  - 感到思考变慢: 立即检查

复杂判断标准:
  - 频繁需要重新读取文件
  - 开始重复相同的问题
  - 越来越难以理解上下文
  - 规则检查通过率下降
  - 响应质量明显下降
```

### Q4: 规则守护程序检查频率应该是多少？

```yaml
正常任务: 每 5 个操作检查一次
复杂任务: 每 3 个操作检查一次
关键操作: 每个操作前检查一次
紧急情况: 连续检查，不限频率

检查触发:
  - 操作计数达到阈值
  - 任务完成阶段
  - 上下文长度升级
  - 感到困惑或不确定
```
