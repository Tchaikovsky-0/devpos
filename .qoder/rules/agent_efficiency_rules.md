# Agent 执行效率规范 - 防止无意义操作

> **核心原则**: 先检查、后行动，避免死循环，高效完成任务
> **版本**: v1.0.0
> **创建日期**: 2026-04-06

---

## 一、问题诊断

### 1.1 典型场景

```yaml
常见无效操作:
  - 反复杀端口 → 重启开发服务器
  - 反复清理缓存 → 重启
  - 反复安装依赖
  - 反复执行相同命令但无进展
  - 无验证就修复，修复后无验证就结束

用户反馈的痛点:
  - "我想看一下web界面，agent一直在杀端口重启"
  - "反复清理缓存，但问题根本不在缓存"
  - "不检查状态就盲目执行"
```

## 二、执行前检查清单（强制执行）

### 2.1 服务状态检查

```yaml
启动服务前必须检查:
  □ 服务是否已经在运行？
  □ 端口是否已被占用？
  □ 如果已在运行，用户为什么还要再启动？

检查命令:
  # 检查端口占用
  lsof -i :<端口号>
  
  # 检查进程状态
  ps aux | grep <进程名>
```

### 2.2 问题定位检查

```yaml
修复问题前必须检查:
  □ 问题现象是什么？
  □ 错误日志在哪里？
  □ 最小复现步骤是什么？
  □ 可能的原因有哪些？（至少列出3个）
  □ 我找到根因了吗？

禁止:
  ❌ 不知道问题就盲目清理缓存
  ❌ 不看日志就猜测原因
  ❌ 不复现就修复
```

### 2.3 缓存问题检查

```yaml
只有满足以下条件才清理缓存:
  ✅ 修改了配置文件但未生效
  ✅ 修改了依赖但未更新
  ✅ 明确的缓存相关错误信息
  ✅ 其他方法都试过了

禁止:
  ❌ 一上来就清理缓存
  ❌ 反复清理缓存
  ❌ 清理后不验证是否解决问题
```

## 三、高效执行流程

### 3.1 启动开发服务器流程

```
用户要求"看一下web界面"或"启动服务"
    ↓
1️⃣ 检查服务是否已在运行
    ↓
    ├─ 已运行 → 直接打开浏览器预览
    │           (不要杀端口重启!)
    │
    └─ 未运行 → 启动服务
                ↓
            等待服务启动成功
                ↓
            打开浏览器预览
```

### 3.2 问题解决流程

```
用户报告问题
    ↓
1️⃣ 收集信息：现象、日志、环境
    ↓
2️⃣ 分析可能原因（至少3个）
    ↓
3️⃣ 逐一验证，排除不可能
    ↓
4️⃣ 定位根因
    ↓
5️⃣ 制定修复方案
    ↓
6️⃣ 实施修复
    ↓
7️⃣ 验证修复效果
    ↓
8️⃣ 确认问题解决，结束
```

## 四、禁止行为清单

### 4.1 绝对禁止

```yaml
🚫 绝对禁止:
  - 不检查状态就杀端口
  - 不分析原因就清理缓存
  - 反复执行相同命令无进展
  - 修复后不验证就结束
  - 用户说"看一下"就反复重启服务

⚠️ 典型错误示例:
  ❌ 用户: "我想看一下web界面"
  ❌ Agent: 杀端口 → 重启 → 杀端口 → 重启...
  ✅ 正确做法: 检查服务状态 → 已运行则直接打开浏览器
```

### 4.2 需要特别注意

```yaml
⚠️ 谨慎操作:
  - 删除文件前必须确认
  - 修改配置前必须备份
  - 大规模操作前必须告知用户
  - 不确定时先问用户
```

## 五、快速参考

### 5.1 状态检查命令

```bash
# 检查前端服务 (假设端口 5173)
lsof -i :5173

# 检查后端服务 (假设端口 8094)
lsof -i :8094

# 查看进程
ps aux | grep -E "node|go|python" | grep -v grep

# 检查开发服务器是否在运行 (Vite)
curl -s http://localhost:5173 | head -20
```

### 5.2 决策树

```
任务开始
    ↓
是启动服务吗？
    ├─ 是 → 检查是否已运行
    │         ├─ 已运行 → 直接使用
    │         └─ 未运行 → 启动
    │
    └─ 否 → 是修复问题吗？
              ├─ 是 → 收集信息 → 分析 → 验证 → 修复 → 验证
              └─ 否 → 正常执行
```

## 六、正确示例 vs 错误示例

### 6.1 启动服务场景

#### ❌ 错误示例
```
用户: "我想看一下前端界面"
Agent: 
  1. lsof -i :5173 → 发现占用
  2. kill -9 <PID>
  3. pnpm dev
  4. 等待...
  5. lsof -i :5173 → 又检查
  6. kill -9 <PID>
  7. pnpm dev
  ... 无限循环
```

#### ✅ 正确示例
```
用户: "我想看一下前端界面"
Agent:
  1. 检查服务状态：lsof -i :5173
  2. 发现服务已在运行 (PID: 12345)
  3. 直接打开浏览器：http://localhost:5173
  4. 告诉用户："服务已经在运行了，我帮你打开浏览器"
```

### 6.2 修复问题场景

#### ❌ 错误示例
```
用户: "页面打不开"
Agent:
  1. rm -rf .vite node_modules/.cache
  2. pnpm install
  3. pnpm dev
  4. 还是打不开
  5. rm -rf .vite node_modules/.cache
  6. pnpm install
  ... 无限循环
```

#### ✅ 正确示例
```
用户: "页面打不开"
Agent:
  1. 收集信息：检查控制台错误、检查服务日志
  2. 发现错误："Module not found: xxx"
  3. 分析原因：依赖未安装
  4. 修复：pnpm install
  5. 验证：刷新页面，确认正常
  6. 结束
```

## 七、小科普：什么是"Agent 死循环"？

🤔 **Agent 死循环**就像：
- 你让机器人去开门
- 机器人走到门前，发现门锁着
- 它不找钥匙，而是反复"推开门 → 发现锁着 → 再推 → 再发现锁着"
- 永远停不下来

🔑 **解决方案**：
- 先观察（检查状态）
- 再思考（分析原因）
- 最后行动（有针对性地解决）

## 八、自动打开浏览器规则

### 8.1 触发条件

```yaml
自动打开浏览器:
  ✅ 前端开发服务器启动后自动打开
  ✅ 预览URL生成后自动打开
  ✅ 本地构建完成后自动打开

不自动打开:
  ❌ API后端服务启动（无UI）
  ❌ 测试运行
  ❌ 脚本执行
```

### 8.2 实现方式

```typescript
// 开发服务器配置
const config = {
  server: {
    open: true  // 自动打开浏览器
  }
};
```
